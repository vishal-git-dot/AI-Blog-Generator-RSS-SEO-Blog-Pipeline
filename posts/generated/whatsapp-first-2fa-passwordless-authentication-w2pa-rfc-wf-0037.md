# WhatsApp-first 2FA Passwordless Authentication (W2PA): RFC-WF-0037

## Overview

Status: Draft Standard Version: 1.0.0 Date: 8 Jun 2026 Category: Standards Track Author: FullAgenticStack Initiative Dependencies: RFC-WF-0001 (WFCS), RFC-WF-0003 (CCP), RFC-WF-0004 (ACSM), RFC-WF-0006 (EAS), RFC-WF-0009 (TMSI), RFC-WF-0010 (IDS), RFC-WF-0014 (CAMN), RFC-WF-0018 (SMCL), RFC-WF-0020 (RAOS), RFC-WF-0021 (SECR), RFC-WF-0023 (TUXP), RFC-WF-0034 (CSCP), RFC-WF-0036 (ETRC) License: Open Specification (Public, Royalty-Free) Abstract This document specifies WhatsApp-first 2FA Passwordless Authentication (W2PA) for systems whose primary operational channel is WhatsApp. W2PA defines the simplest safe authentication pattern for WhatsApp-first systems: the user either enters their phone number on the system login page or sends an unauthenticated message to the system's WhatsApp number. In both cases, the system sends a single-use magic-link through WhatsApp. When opened, the link displays only one action button, Entrar . When clicked, the page initiates a passkey ceremony. After successful passkey verification, the system authenticates the actor, confirms login through WhatsApp when the flow started on the web, or executes the originally requested WhatsApp command when the flow started from an unauthenticated message. W2PA intentionally does not use passwords and does not require e-mail as a primary authentication channel. The WhatsApp channel proves possession of the phone route; the passkey proves possession of the user's cryptographic authenticator. The magic-link is not sufficient to authenticate the user by itself. It is a short-lived transport binding that starts the passkey ceremony. Index Terms— WhatsApp-first, passwordless authentication, passkeys, magic-link, 2FA, WebAuthn, login confirmation, registerless authentication, command continuation. I. Introduction WhatsApp-first systems should not inherit the complexity of web-first authentication flows when the user's primary interaction channel is already WhatsApp. A traditional flow asks the user to remember a password, receive an e-mail, open a dashboard, or complete a registration form before doing the thing they wanted to do. That pattern is not aligned with WhatsApp-first operation. W2PA defines a simpler model: The user proves control of the WhatsApp route by receiving a magic-link in WhatsApp. The user proves strong possession with a passkey. The system either opens an authenticated web session or continues the pending WhatsApp command. The user-visible page opened by the magic-link MUST be minimal. It MUST NOT present registration forms, password fields, e-mail fallback, account recovery menus, marketing pages, or dashboard content. Its primary action is one button: Entrar . The button is used for both first access and later access. If no passkey credential exists for the phone-bound actor, the passkey ceremony creates one under policy. If a credential already exists, the passkey ceremony authenticates the actor. This makes registration and login the same user action: the user enters. II. Scope W2PA specifies: the two canonical initiation surfaces for WhatsApp-first authentication; magic-link creation, delivery, expiry, and binding requirements; the minimal Entrar page behavior; passkey registration/authentication behavior; authenticated continuation for web-started and WhatsApp-started flows; evidence, idempotency, lifecycle, error, and abuse-control requirements; required conversational outputs after authentication. W2PA does not define a full WebAuthn profile, passkey vendor, WhatsApp provider, browser implementation, or cryptographic library. It defines protocol obligations and observable behavior for WhatsApp-first compliance. III. Normative Language MUST , MUST NOT , SHOULD , SHOULD NOT , and MAY are normative. IV. Definitions W2PA: WhatsApp-first 2FA Passwordless Authentication. Magic-link: A short-lived, single-use URL delivered through WhatsApp that opens the minimal authentication page and binds a pending authentication challenge to a phone-route proof. Entrar Page: The minimal page opened by the magic-link. It displays only the authentication action necessary to continue: Entrar . Passkey Ceremony: A WebAuthn-compatible credential creation or assertion ceremony used to create or verify a passkey. Phone-bound Actor: The actor identity candidate derived from the WhatsApp sender or from a phone number entered on the login page. Unauthenticated Command Continuation: A pending WhatsApp request stored before authentication and executed after W2PA succeeds. Login Confirmation Message: A WhatsApp message sent after successful authentication to confirm that a login or command continuation occurred. V. Design Goals W2PA MUST ensure: G1. Simplicity: the user authenticates by receiving a WhatsApp link and pressing Entrar . G2. Passwordless operation: no password field is required or accepted in the W2PA flow. G3. Phishing-resistant verification: successful authentication requires passkey verification, not only possession of the magic-link. G4. WhatsApp-first continuity: a command sent while unauthenticated resumes after authentication instead of being discarded. G5. Registerless UX: first access and login use the same user-facing action. G6. Auditability: every authentication transition emits evidence. G7. Replay resistance: duplicated links, duplicated messages, or repeated clicks cannot create multiple sessions or repeated command effects. G8. Safe notification: successful web authentication is confirmed through WhatsApp. VI. Canonical Authentication Initiation Surfaces W2PA defines exactly two canonical initiation surfaces. A. Web Login Initiation The system MAY expose a web login page containing a phone input and a submit action. The login page MUST accept a phone number as the primary identifier. The system MUST normalize the phone number into the canonical actor lookup format before creating a challenge. After the user submits the phone number, the system MUST send the magic-link to the corresponding WhatsApp route. The web page SHOULD show only a waiting state, such as: “Open the link sent to your WhatsApp.” The web page MUST NOT authenticate the user merely because the phone number was submitted. B. WhatsApp Unauthenticated Message Initiation If a user sends a message to the system's WhatsApp number without an authenticated context, the system MUST treat the message as a pending unauthenticated request. The system MUST: normalize the inbound message via CAMN; create a pending authentication context via CSCP; store the original request with trace bindings; send a magic-link to the same WhatsApp conversation; after successful W2PA, execute the original request if it is still valid and authorized. The system MUST NOT discard the user's original request merely because authentication was required. VII. Magic-link Requirements A. Link Purpose The magic-link MUST only start or resume the W2PA ceremony. It MUST NOT, by itself, create an authenticated session, execute a command, grant scopes, register credentials, or bypass passkey verification. B. Delivery Channel The magic-link MUST be delivered through WhatsApp to the phone-bound actor. E-mail MUST NOT be required for the normal W2PA flow. Alternative recovery channels MAY exist, but they are outside the normal W2PA path and MUST be governed by ACSM, TMSI, RAOS, and evidence requirements. C. Single-use and Expiry A magic-link MUST be: single-use; short-lived; bound to an authentication challenge; bound to the phone-bound actor; bound to a tenant and system environment; bound to a flow type: web_login or whatsapp_continuation ; invalidated after successful passkey completion, explicit cancellation, expiry, or abuse detection. A system SHOULD use expiry windows measured in minutes, not hours. D. Opaque Token The URL token MUST be opaque and unguessable. It MUST NOT expose raw phone numbers, actor IDs, scopes, command payloads, or passkey credential identifiers. E. Link Reuse Behavior If a consumed or expired link is opened, the system MUST NOT authenticate the user. It MUST show a safe failure and MAY offer to request a new link through WhatsApp. The failure MUST emit evidence and SHOULD use an ETRC reason code such as WFS.AUTH.LINK_EXPIRED or WFS.AUTH.LINK_ALREADY_USED . VIII. Entrar Page Requirements A. Minimal Surface The page opened by the magic-link MUST present only the authentication continuation necessary for W2PA. The canonical UI is: Entrar The localized label MAY vary by language, but the semantic action MUST remain “enter/authenticate/continue.” B. Forbidden Elements The Entrar Page MUST NOT include: password fields; e-mail login fields; account registration forms; profile completion forms; dashboard content; operational command controls; admin controls; unrelated navigation; OAuth provider buttons unless explicitly governed as an extension profile. C. Button Behavior When the user clicks Entrar , the page MUST initiate a passkey ceremony. If the phone-bound actor has no registered passkey credential and policy permits first access, the system MAY initiate passkey creation. If the actor already has a credential, the system MUST initiate passkey authentication. A system MUST NOT create a persistent authenticated session until the passkey ceremony succeeds. D. No Separate Register Button W2PA SHOULD NOT expose a separate “Register” button. First access and login are the same operation from the user's perspective. The system MAY internally distinguish passkey enrollment from passkey authentication, but the user-facing action remains Entrar . IX. Passkey Ceremony Requirements A. Required Authentication Strength W2PA requires passkey-based verification as the second factor. The WhatsApp-delivered magic-link proves access to the phone route; the passkey proves possession of a cryptographic authenticator. A successful W2PA session SHOULD be classified at least as ACSM L2 . Privileged commands MAY still require L3 step-up. B. Credential Binding Passkey credentials MUST be bound to a stable actor identity, tenant context, and policy-defined relying party identity. The system MUST NOT bind passkeys only to a raw mutable display phone string. It SHOULD bind to an internal actor ID derived from verified phone-route ownership. C. First Credential Enrollment When no passkey exists for the actor, the system MAY allow first credential enrollment only if: the challenge was initiated through the actor's WhatsApp route; the magic-link is valid and single-use; the passkey ceremony succeeds; enrollment policy allows registerless first access; evidence is emitted for the enrollment decision. D. Existing Credential Authentication When a passkey exists, the system MUST require a valid assertion from an accepted credential before issuing the authenticated context. E. Multiple Credentials The system MAY support multiple passkeys per actor. Adding, removing, or resetting credentials MUST be treated as a recovery or security-sensitive operation and MUST follow ACSM, TMSI, RAOS, and evidence requirements. X. Web-started Flow A web-started W2PA flow begins when the user enters a phone number on the login page. A. Required Sequence User enters phone number on the login page. System normalizes the phone number. System creates an authentication challenge. System sends a magic-link to the user's WhatsApp. User opens the magic-link. The Entrar Page is displayed. User clicks Entrar . Browser performs passkey ceremony. System verifies the passkey result. System creates an authenticated web session. System sends a WhatsApp login confirmation message. B. Login Confirmation Message After successful web authentication, the system MUST send a WhatsApp confirmation message to the authenticated actor. The confirmation message MUST include: that login was confirmed; the system name or tenant name; approximate time; client metadata summary sufficient for user recognition, redacted by policy; an option to report if it was not the user. Example structure: Login confirmado. Sistema: <system_name> Horário: <timestamp> Origem: <client_summary> Se não foi você, responda: 1 - Bloquear sessão The system MUST NOT include secrets, raw tokens, full IP addresses, or credential identifiers in the confirmation message. XI. WhatsApp-started Unauthenticated Command Flow A WhatsApp-started W2PA flow begins when an unauthenticated user sends a request to the system's WhatsApp number. A. Required Sequence User sends a message to the system while unauthenticated. System normalizes the message via CAMN. System stores a pending context via CSCP. System creates an authentication challenge bound to the pending request. System sends a magic-link in the same WhatsApp conversation. User opens the magic-link. The Entrar Page is displayed. User clicks Entrar . Browser performs passkey ceremony. System verifies the passkey result. System resumes the pending request. System canonicalizes the request into CCP if it is executable. System applies authorization, confirmation, approval, idempotency, and lifecycle rules. System responds in WhatsApp with the command result or next required step. B. Pending Request Validity The pending request MUST have an expiry. If the request expires before authentication succeeds, the system MUST NOT execute it. It SHOULD ask the user to send the request again. C. Command Execution After Authentication After authentication succeeds, the system MUST NOT blindly execute the original raw text. It MUST continue through normal WhatsApp-first command processing: canonicalization into a CCP envelope; safety classification through CRCD/DMSG rules; confirmation for mutations; strengthened confirmation for destructive actions; ACSM scope and step-up checks; IDS idempotency handling; SMCL lifecycle transitions; EAS evidence emission. If the original request is read-only and authorized, the system MAY execute and return the result immediately. If the original request is mutating, destructive, ambiguous, or high-impact, the system MUST ask for the required confirmation, disambiguation, approval, or step-up after authentication. XII. Session and Continuation Binding A. Auth Context A successful W2PA ceremony MUST create an auth_context_id that can be referenced by CCP, ACSM, EAS, CSCP, and OoC. The auth context MUST include at minimum: actor ID; tenant ID; trust level; authentication method: whatsapp_magic_link_plus_passkey ; creation time; freshness expiry; device/session binding metadata; flow type: web_login or whatsapp_continuation . B. Browser Session Binding For web sessions, the system SHOULD bind the session to a proof-of-possession mechanism such as DPoP, mTLS, or another policy-approved channel binding mechanism. C. Linear Session Destruction If the implementation uses LinearAutoDestroy keys or equivalent ephemeral session material, the system SHOULD destroy one-time authentication secrets immediately after successful verification, expiry, or rejection. D. Context Closure After the flow reaches a terminal state, the W2PA context MUST be closed or expired. It MUST NOT remain reusable for later authentication or command continuation. XIII. Risk Signals and Non-blocking Location Changes The system SHOULD collect risk signals for authentication decisions, including: user agent family; platform family; coarse network metadata; coarse geolocation or region when available; device binding state; challenge age; link open/client mismatch; repeated failed passkey attempts; impossible-travel-like anomalies. The system MUST treat risk signals as policy inputs, not as sole proof of compromise. The system MUST NOT block a legitimate user solely because the apparent location changed. Users travel, networks change, VPNs exist, and mobile carriers route traffic unpredictably. Location discrepancy MAY trigger additional confirmation, login notification, temporary lower trust level, or operator-visible evidence, but it MUST NOT be the only reason for permanent lockout. The system MAY allow the user to declare travel mode through WhatsApp without requiring the user to disclose the destination. XIV. Evidence Requirements W2PA-compliant systems MUST emit EAS-compatible evidence for: authentication challenge created; magic-link sent; magic-link opened; Entrar button clicked; passkey ceremony started; passkey ceremony succeeded or failed; passkey credential enrolled, when applicable; auth context created; web login confirmation sent; unauthenticated request resumed; pending request expired or rejected; session blocked, revoked, or reported. Evidence artifacts SHOULD use extension fields under SECR-compatible extensions.x_w2pa when W2PA-specific fields are needed. A. Suggested W2PA Evidence Extension { "extensions" : { "x_w2pa" : { "flow_type" : "web_login" , "challenge_id" : "auth_chal_123" , "magic_link_id" : "ml_123" , "passkey_operation" : "authentication" , "auth_context_id" : "authctx_123" , "pending_request_id" : null , "client_risk_summary" : { "risk_level" : "low" , "signals" : [ "known_device" , "fresh_challenge" ] } } } } XV. Lifecycle Model W2PA authentication challenges SHOULD expose a lifecycle compatible with SMCL semantics. A. Required States auth_received — phone/web or WhatsApp unauthenticated request received. challenge_created — authentication challenge created. link_sent — magic-link sent through WhatsApp. link_opened — magic-link opened. passkey_required — Entrar Page is ready to start passkey ceremony. passkey_started — browser ceremony started. authenticated — passkey verified and auth context created. continued — pending web session or WhatsApp command continuation completed. failed — authentication failed. expired — challenge expired. revoked — challenge or session revoked. B. Terminal States authenticated , continued , failed , expired , and revoked are terminal for the challenge. A terminal challenge MUST NOT be reused. XVI. Idempotency and Replay Requirements A. Duplicate Web Submit Repeated submission of the same phone number within a short window SHOULD converge to a single active challenge or intentionally rotate the challenge while revoking the previous one. B. Duplicate WhatsApp Message Duplicate inbound messages MUST NOT create duplicate pending commands. CAMN and IDS MUST be used to deduplicate message deliveries. C. Duplicate Link Opens Opening the same magic-link multiple times MUST NOT create multiple authenticated contexts. If a valid challenge is already in progress, the system MAY show the same Entrar Page until the challenge is consumed or expired. D. Duplicate Passkey Completion Duplicate passkey completion callbacks MUST be idempotent. If authentication has already succeeded, the system MUST replay the terminal outcome and MUST NOT create additional sessions unless policy explicitly allows multiple sessions and each is separately evidenced. E. Duplicate Command Continuation For WhatsApp-started flows, resuming a pending command MUST use IDS. The original request MUST NOT execute more than once because the magic-link was opened or passkey callback was delivered more than once. XVII. Abuse Controls W2PA implementations MUST enforce RAOS-aligned controls for: phone submission rate; magic-link issuance rate; link open attempts; passkey failure attempts; challenge creation per actor/tenant; pending unauthenticated request storage; repeated unauthenticated command attempts; repeated session report/lock actions. Confirmation, passkey, and link failures SHOULD use generic user-facing messages while evidence records preserve precise reason codes. The system MUST NOT reveal whether a phone number exists in the system through different public responses on the web login page. It SHOULD use a neutral response such as: “If this number can access the system, a link was sent through WhatsApp.” XVIII. Error and Reason Codes W2PA extends ETRC with the following recommended reason codes: WFS.AUTH.PHONE_INVALID — phone number cannot be normalized. WFS.AUTH.LINK_SENT — challenge link sent. WFS.AUTH.LINK_EXPIRED — magic-link expired. WFS.AUTH.LINK_ALREADY_USED — magic-link already consumed. WFS.AUTH.LINK_REVOKED — challenge revoked before use. WFS.AUTH.PASSKEY_REQUIRED — passkey ceremony required. WFS.AUTH.PASSKEY_FAILED — passkey verification failed. WFS.AUTH.PASSKEY_NOT_REGISTERED — no passkey exists and enrollment is not allowed. WFS.AUTH.PASSKEY_ENROLLED — first credential enrolled. WFS.AUTH.AUTHENTICATED — authentication succeeded. WFS.AUTH.PENDING_REQUEST_EXPIRED — original WhatsApp request expired before authentication. WFS.AUTH.CONTINUATION_REJECTED — pending request cannot be resumed. WFS.AUTH.RISK_STEPUP_REQUIRED — risk policy requires extra verification. WFS.AUTH.SESSION_REPORTED — user reported a login as suspicious. Implementations MAY add domain-specific codes using SECR extension rules. XIX. Conversational Templates A. Suggested Capabilities auth.passwordless.start_web auth.passwordless.start_whatsapp auth.passwordless.verify_passkey auth.passwordless.confirm_login auth.passwordless.continue_pending_request auth.passwordless.report_session B. Suggested Commands AUTH.START_WEB_LOGIN AUTH.START_WHATSAPP_LOGIN AUTH.VERIFY_PASSKEY AUTH.CONTINUE_PENDING_REQUEST AUTH.REPORT_SESSION AUTH.REVOKE_SESSION Session revocation and credential reset MUST be classified as security-sensitive or recovery operations and governed by ACSM/RAOS. XXI. Security Considerations A. Magic-link Is Not Authentication A magic-link is a delivery-bound challenge handle. It MUST NOT be treated as sufficient authentication. B. WhatsApp Route Is a Factor, Not the Whole Identity Receiving WhatsApp messages indicates control of the phone route at that moment. It does not prove long-term human identity by itself. W2PA requires passkey verification to complete authentication. C. Passkey Reset Is High Risk Credential reset, adding backup credentials, or removing passkeys can transfer account control. These operations MUST be treated as high-impact recovery actions. D. Account Enumeration Public web login responses MUST be neutral. The system MUST NOT reveal whether a phone number is registered or allowed. E. Device Loss If the user loses the phone or authenticator, recovery MUST follow governed recovery procedures. Recovery MUST NOT silently downgrade to password or e-mail-only authentication unless explicitly declared as a separate weaker profile. F. Session Reporting Login confirmation messages MUST provide a way to report suspicious access. Reporting MUST trigger governed mitigation, such as session revocation, temporary lock, step-up escalation, or operator review. XXII. Privacy Considerations W2PA systems MUST minimize authentication telemetry exposed to the user and stored in evidence. Login confirmation messages SHOULD include enough metadata for recognition but MUST avoid sensitive detail. For example, coarse device/browser summary is acceptable; full IP address, exact geolocation, internal risk scores, and credential IDs SHOULD NOT be shown to normal users. Evidence MAY store more detailed risk metadata if governed by DRPC, redaction policies, and privileged access controls. XXIII. Conformance Requirements A system is W2PA-compliant if it satisfies all requirements below: It supports web login initiation by phone number. It supports unauthenticated WhatsApp message initiation. It sends magic-links through WhatsApp. A magic-link alone cannot authenticate the user. The magic-link opens a minimal Entrar Page. The Entrar Page initiates passkey registration or authentication. Successful authentication creates a governed auth context. Web-started login sends a WhatsApp login confirmation. WhatsApp-started unauthenticated requests resume after authentication. Mutating resumed requests still require CCP confirmation. Destructive or high-impact resumed requests still require strengthened confirmation, approval, or step-up as applicable. All major transitions emit evidence. Duplicate messages, duplicate link opens, and duplicate passkey callbacks are idempotent. Abuse controls exist for link issuance and passkey attempts. Public responses do not leak account existence. XXIV. Reference Test Scenarios W2PA-TVRS-001 — Web Login Success Input: user enters phone number on web login page.
