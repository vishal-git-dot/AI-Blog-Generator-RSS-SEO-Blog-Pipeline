# Spark Connect: Decoupled Client-Server Architecture, gRPC & Multi-Language Sessions

## Overview

spark connect is the single biggest architectural change in Apache Spark since DataFrames replaced RDDs — and it quietly rewires every assumption senior data engineers carry from years of spark-submit , fat driver JARs, and JVM-coupled clients. The classic Spark deployment is a monolith: the driver, the SparkContext, the user code, the catalyst optimiser, and the executor scheduler all run together in one JVM, and every client language (PySpark, sparklyr, Spark R) has to wrestle with that JVM through Py4J, RPC bridges, or shell processes. Spark Connect breaks that monolith into a thin client and a remote SparkSession reachable over spark grpc , and the consequences ripple through every part of the stack — language clients, IDE integration, notebook fleets, multi-tenant platforms, and the API surface itself. Why Spark Connect changed the deployment model Spark Connect splits the monolithic driver into a thin client + a remote SparkSession — the protocol, not the DSL, is what makes the engineering trade-off The one-sentence invariant: Spark Connect replaces the single-JVM Spark driver with a thin client (any language, no JVM required) that talks to a remote Spark Connect Server over gRPC, sending unresolved logical plans as protobuf and receiving Arrow result batches back . Every other consequence — language clients, IDE integration, session isolation, the API gap list, deployment shape — follows from that one structural change. Once you internalise "thin client, remote SparkSession, gRPC on the wire," the entire Spark Connect interview surface collapses to a sequence of consequences from that one decoupling. Four axes interviewers actually probe. Protocol. What is on the wire between the client and the server? Spark Connect serialises an unresolved logical plan as a protobuf message, ships it over gRPC, and streams the result back as Arrow record batches. The protocol is the contract; everything else — Python client, Scala client, Go client — is a different implementation of the same protocol. Client. What is a Spark Connect client? In the Python case, it is a pure-Python library — no JVM, no Py4J gateway, no SPARK_HOME on the client machine, no fat client install. The client builds a DataFrame, lazily, by appending to a protobuf plan. When you call .collect() or .show() , that plan is sent to the server. Session. What is a Spark Connect session? A logical handle, identified by a session ID and a user ID, that the Connect Server maps to its own SparkSession, its own temp views, its own UDFs, and its own SQL config. Multiple clients can share a Connect Server; each one gets its own session and the server isolates them. Isolation. What separates two sessions on the same server? Catalog, temp views, SQL config, registered UDFs, broadcast variables — all per session. Compute (CPU, memory) is not fully isolated by default — sessions share the executor pool — and that is the single biggest production gotcha senior engineers have to design around. Why the decoupling matters in 2026. Lifecycle independence. A spark-submit job's driver lives and dies with the job. A Spark Connect client lives in your IDE, your notebook kernel, or your microservice; the server lives in a long-running cluster. Restarting the client does not kill the cluster. Language portability. Every non-JVM client (PySpark, sparklyr, Spark R) has been a clever bridge — Py4J, R-on-JVM, or process-level RPC. Spark Connect replaces every bridge with one gRPC protocol; a new language client is a code-generation job from the protobuf definitions, not a JVM-integration project. Cluster footprint. With Spark Connect, you can ship a 10 MB Python wheel ( pyspark[connect] ) to your laptop, your CI, or your Lambda instead of a 400 MB Spark distribution. IDE / notebook fleet. A notebook server now hosts hundreds of thin client processes against one Connect Server; the notebook server holds no Spark state. GA + hardening timeline — what to know in interviews. Spark 3.4 (2023) shipped Connect as GA. Spark 3.5 narrowed the API gap (Pandas UDFs, streaming, more catalog ops). Spark 4.0 locked in the protocol as the primary deployment mode, with richer SQL, broader streaming, and improved Arrow encoding. In 2026 most managed Spark offerings expose a Connect endpoint by default; greenfield Spark 4 pipelines are increasingly Connect-first, with spark-submit reserved for batch ETL and RDD legacy. In the classic path, the local JVM holds the SparkContext, the Catalyst optimiser, the scheduler, and the broadcast table for closures. RDD and DataFrame operations build a logical plan inside the local JVM, which then ships RDD partitions to executors as Java-serialised closures. In the Connect path, the local Python process builds an unresolved logical plan as a protobuf object. There is no JVM on the client side — pyspark[connect] is a pure-Python wheel. When .show() is called, the entire plan is shipped to the server in one gRPC ExecutePlan request. The server receives the plan, runs Catalyst (analysis → optimisation → physical planning) on its own SparkSession, executes the job on its existing executor pool, and streams Arrow batches back over gRPC for the client to format as a .show() table. In the classic shape, the notebook kernel literally hosts the SparkContext JVM. Killing the kernel kills the JVM, which tears down the SparkSession, releases all cached partitions, and returns executor slots to the cluster scheduler. A reconnecting client that knows the original session ID can pick up where the previous one left off — including cached DataFrames. On cluster restart (the Connect Server process restarts), all in-memory session state is lost. The next client request creates a new session. Long-running pipelines should therefore not rely on session-cached state surviving a server restart; persist intermediates to a durable store or accept the rebuild cost. The idle-timeout config ( spark.connect.session.timeout ) controls how long an idle session persists after the last client disconnect. Default values are tuned for interactive workloads (tens of minutes); platform teams running shared servers tune this to match cost and capacity. The net effect: client crashes become cheap (restart the notebook, reattach the session), but session state still has bounded lifetime. The Spark Connect protocol spark connect protocol ships unresolved logical plans as protobuf over gRPC — the wire format, not the DSL, is the contract The mental model in one line: the Spark Connect protocol defines a gRPC service whose request payload is an unresolved logical plan serialised as protobuf, and whose response is a stream of Arrow record batches encoding the query result . Once you say "protobuf plan in, Arrow batches out, gRPC carries both," every other protocol question — multi-language clients, plan visibility, streaming results, error handling — becomes a deduction from that contract. The protocol surface — what gRPC services are defined. SparkConnectService — the core service. Main RPCs: ExecutePlan , AnalyzePlan , Config , AddArtifacts , Interrupt , ReattachExecute , ReleaseExecute . ExecutePlan — server-streaming RPC. Client sends one ExecutePlanRequest with the protobuf plan; server streams back ExecutePlanResponse messages, each carrying one Arrow record batch or a control message. AnalyzePlan — unary RPC for .printSchema() , .explain() , .columns — schema and plan inspection without execution. Config — unary RPC for spark.conf.get / set / unset . Per-session config lives on the server. AddArtifacts — bidirectional streaming. Uploads JARs, Python files, archives into the session classpath. Interrupt — unary RPC to cancel an in-flight ExecutePlan (the Ctrl+C story). Anatomy of an unresolved logical plan. The plan is a protobuf message named Relation . Every DataFrame operation in the client builds a tree of Relation nodes: Read , Project , Filter , Aggregate , Join , Sort , Limit , Sample , WithColumn , etc. Each Relation carries its operands (sub-relations) and its parameters (column expressions, predicates, join conditions). Expressions are themselves a tree ( Expression protobuf with variants like Literal , UnresolvedAttribute , UnresolvedFunction , Cast , BinaryOp ). "Unresolved" means the client has not bound column names to actual schema fields or function names to actual implementations. That binding happens server-side during the Analyzer phase of Catalyst. This is critical: the client does not need a schema to build a plan. It can reference df["user_id"] without knowing whether user_id is a string or a long — the server resolves it when the plan is executed. The execution path on the server. The server receives the protobuf plan and constructs an internal LogicalPlan from it (the same LogicalPlan data structure the classic Spark driver uses). The plan flows through Catalyst: Analyzer → CacheManager → Optimizer → SparkPlanner → physical plan with AQE rules. The physical plan executes on the server's existing executor pool. Arrow record batches are streamed back to the client via the ExecutePlan response stream. The session has its own SQL config, its own temp views, its own UDFs, its own catalog cache. The server multiplexes many sessions on top of the same executor pool. The Arrow result format — why batches matter. Results are streamed as Apache Arrow record batches. Each ExecutePlanResponse carries either zero or one Arrow batch in its arrow_batch field. Arrow's columnar format is portable across languages — the Python client deserialises directly into a pyarrow.Table or pandas.DataFrame ; the Scala client gets a Dataset[Row] ; the Go client gets an Arrow record reader. Batch sizes are configurable via spark.connect.grpc.arrow.maxBatchSize (default ~4 MB). Larger batches reduce gRPC framing overhead; smaller batches reduce time-to-first-byte for .show() previews. The Arrow stream may also carry a Metrics message at the end (query metrics, plan execution time, executor stats) and an ExecutionProgress message during long-running queries (for .show_progress() -style UI). Plan introspection — what the client can see locally. A Connect client can call .explain(mode="extended") which sends an AnalyzePlan request. The server returns the analysed plan, the optimised plan, and the physical plan as protobuf, and the client formats them back to text locally. A Connect client can call .printSchema() which sends an AnalyzePlan request with mode SCHEMA . The server returns the schema protobuf; the client formats it. The client cannot see the executor topology, the partition count, or per-task statistics directly — those live on the server. For deep observability, point a Spark UI or a metrics scraper at the server. Common interview probes on the protocol. "What is on the wire between a Spark Connect client and the server?" — unresolved logical plan as protobuf, Arrow batches back. "What does the server do with the plan?" — runs Catalyst (Analyzer → Optimizer → SparkPlanner) and executes on its own executor pool. "Why protobuf for the plan?" — language-neutral, schema-evolution-safe, efficient serialisation. Lets any language with a protobuf code generator become a client. "Why Arrow for the result?" — columnar, language-neutral, zero-copy between server JVM and Python / Go / Rust clients via Arrow IPC. The client sends an ExecutePlanRequest to the server's ExecutePlan gRPC method. This is a server-streaming RPC: one request, many responses. The server keeps the stream open until the query completes (or is interrupted). The server begins executing the query. As executor tasks finish and return results, the server's connect handler accumulates them into Arrow record batches sized to spark.connect.grpc.arrow.maxBatchSize (default ~4 MB). Each batch becomes one ExecutePlanResponse message. The server flushes the message to the gRPC channel; HTTP/2 frames carry it to the client. The server never holds the full result in memory — backpressure is provided by gRPC flow control (the client must consume batches as fast as it can or the server will pause). The client receives each ExecutePlanResponse , deserialises the Arrow batch, and either accumulates it ( .collect() builds a full table) or yields it to user code ( .toLocalIterator() streams batch-by-batch). Session isolation + multi-tenancy A shared spark connect server multiplexes many sessions onto one cluster — catalog and configs isolate; compute does not (by default) The mental model in one line: a Spark Connect Server hosts many SparkSessions concurrently, each identified by (user_id, session_id) , each with its own catalog and configs but sharing the underlying executor pool — so logical isolation is first-class while compute isolation requires explicit scheduler-pool design . Once you say "logical per session, compute shared by default," every multi-tenancy interview question becomes a sequence of consequences from that split. Session identity and lifecycle. A new client call to SparkSession.builder.remote(...).getOrCreate() opens a gRPC channel and sends a Config or ExecutePlan request with user_id and session_id headers. If the (user_id, session_id) pair is new, the server creates a fresh SparkSession via SparkSession.cloneSession() (Spark's built-in session cloning facility) — fast, in-process, no new driver JVM. The session persists in the server's session registry until either an explicit Close (rare) or the configured idle timeout passes. Reconnecting a client with the same (user_id, session_id) finds the same session, with its caches, temp views, and configs intact. What is isolated per session. Temp views — df.createOrReplaceTempView("v") registers v in the session catalog only. Other sessions cannot see it. SQL config — spark.conf.set("spark.sql.shuffle.partitions", "200") is a per-session knob; doesn't leak. Catalog current database / schema — spark.catalog.setCurrentDatabase("analytics") is per-session. Registered UDFs — spark.udf.register("my_fn", ...) registers in the session's UDF registry, not the server-wide one. Cached DataFrames — df.cache() caches relative to the session; other sessions cannot reuse the same logical plan's cache by name. What is shared across sessions (by default). Executor pool — all sessions on a Connect Server share the underlying executor allocation. A heavy query in one session contends for executor slots with all other sessions. Shared catalog (Hive Metastore, Iceberg REST catalog, Unity Catalog) — the persistent catalog is server-wide; sessions read and write the same tables. Server-level SparkConf — spark.executor.memory , spark.driver.memory , network configs — these are set at server boot and apply globally. Cluster scheduler resources — slots, dynamic allocation, queues. By default, sessions compete in the same FIFO scheduler pool. Compute isolation patterns — making sessions fair. Spark Scheduler Pools. Configure spark.scheduler.allocation.file to define named pools (e.g. interactive , batch , etl ) with fair-scheduler weights. Each session can opt into a pool via spark.scheduler.pool . Heavy batch users can be isolated from interactive analysts. Dynamic allocation per pool. With Kubernetes-managed Spark, each pool can have its own min/max executor cap. Combined with Spark's dynamicAllocation , you get loose CPU isolation. Server per tenant. For strict isolation, run one Spark Connect Server per tenant (one per team or per business unit). Trades operational cost for clean isolation. Common for enterprise / regulated workloads. Server per workload class. Run a connect-interactive server for notebooks and a connect-etl server for scheduled batch. Each has its own cluster sizing and queue depth. Authentication and authorization patterns. TLS at the gRPC layer. Spark Connect supports TLS via spark.connect.grpc.server.useTransportSecurity=true and certificate config. Required for any production deployment outside a fully trusted network. Interceptor-based auth. The Connect Server supports custom gRPC interceptors via spark.connect.authenticate.token (token-based) or a custom Java interceptor class. Used to validate bearer tokens (OAuth, JWT) or mTLS client certs. User identity propagation. The interceptor sets the user_id on the session, which downstream Spark code can read for catalog ACL enforcement (e.g., Unity Catalog, Apache Ranger). Server-side user impersonation. For Hive Metastore and HDFS access, the server can impersonate the client user via Hadoop's UserGroupInformation — preserving filesystem ACLs. The "shared pool, fair scheduler" recipe (most common production pattern). Configure spark.scheduler.mode=FAIR at server boot. Define a fairscheduler.xml with named pools ( interactive , etl , default ). Per client, set spark.scheduler.pool via spark.conf.set(...) early in the session. The fair scheduler now balances tasks across pools by weight; heavy ETL queries no longer starve interactive notebooks. Common interview probes on multi-tenancy. "What is isolated between two sessions on a Connect Server?" — catalog/temp views/configs/UDFs/caches per session. Executor pool shared. "How do you enforce CPU isolation between sessions?" — Spark scheduler pools + fair scheduler + per-pool dynamic allocation. "When would you run one server per tenant?" — regulated workloads, strict cost attribution, when one tenant's heavy query risk would block others under fair scheduling. "How do clients authenticate to the Connect Server?" — TLS + token interceptor (custom gRPC interceptor); user identity propagated to Spark session for catalog ACLs. At boot, the Connect Server reads fairscheduler.xml and registers the two named pools. Each pool has a weight (how much of the executor pool it gets when contended) and a minShare (the minimum number of slots reserved for it). On each ExecutePlan request, the server looks up the calling session's spark.scheduler.pool config. If set to interactive , tasks for that query go into the interactive pool's task queue. Within a pool, scheduling is either FAIR (round-robin per job) or FIFO (first-come-first-served). interactive uses FAIR so notebook users do not starve each other; etl uses FIFO so each ETL job runs to completion in order. Migration + production patterns Migrating to spark 4 connect is mostly two lines — but the API-gap surface and deployment shape are where senior engineers earn their keep The mental model in one line: migrating a PySpark job from classic to Connect is .master(...) → .remote(...) plus a forbidden-API audit; deploying a Spark Connect Server in production is "long-running JVM with TLS-fronted gRPC, dynamic executor allocation, observability piped to Prometheus, and a fair-scheduler config" . Once you say "two-line migration + API-gap list + long-running server," every production-pattern interview becomes a deduction from those pieces. The migration playbook — order matters. Step 1: Audit forbidden APIs. Grep the codebase for sc.parallelize , sc.accumulator , sc.broadcast , .rdd , JavaRDD , JavaSparkContext . Anything that hits has to be rewritten or stays on classic. Step 2: Audit UDF surface. Plain Python UDFs and Pandas UDFs migrate cleanly. Scala UDFs require uploading a JAR via AddArtifacts (or pre-installing on the server). Java-class-based custom aggregators may not have direct Connect equivalents. Step 3: Swap .master(...) for .remote(...) . Two lines per script. The rest of the code is identical. Step 4: Run integration tests against a local Connect Server. pyspark[connect] ships a local server that runs in-process for tests, eliminating "what if this works on classic but not Connect" surprises. Step 5: Deploy a real Connect Server. Kubernetes / EMR / Databricks all support Spark Connect Server in 2026. Pick the simplest managed offering you have. Step 6: Roll out client by client. Forbidden API checklist (the ones you actually hit). sc.parallelize(...) — replace with spark.createDataFrame(...) . Same end state for small synthetic data. sc.accumulator(...) — rewrite the metric using a SQL aggregation. Accumulators were always a niche pattern; the DataFrame replacement is cleaner. sc.broadcast(...) — for small lookup tables, use broadcast(df) from pyspark.sql.functions . The hint instructs Catalyst to do the broadcast join. Direct broadcast-variable use is gone. df.rdd — most uses of .rdd are for old map / flatMap / reduceByKey patterns. Frequently asked questions What is Spark Connect and why does it matter? spark connect is the decoupled client-server architecture introduced in Apache Spark 3.4 and hardened through Spark 4.x. It replaces the monolithic JVM driver — where the SparkContext, the Catalyst optimiser, the user code, and the executor scheduler all run in a single process — with two cleanly separated pieces: a thin client (any language with a gRPC + protobuf stub) that builds unresolved logical plans, and a Spark Connect Server that hosts long-running SparkSession s, runs Catalyst + execution on its own executor pool, and streams Arrow record batches back. It matters because it changes the deployment shape: notebooks, IDEs, Lambdas, and microservices can now drive Spark queries without embedding a JVM, multi-language teams converge on one protocol, and session lifecycle is independent of client lifecycle (the kernel can crash and reconnect without losing cached state). For senior interviews in 2026, "Spark Connect is a thin client + remote SparkSession over gRPC" is the load-bearing one-sentence answer. Does Spark Connect work with PySpark? Yes — pyspark connect is the most polished client in the matrix. pip install pyspark[connect]==4.0.0 installs a slim (~10 MB) pure-Python wheel with no JVM dependency. Code looks essentially identical to classic PySpark: from pyspark.sql import SparkSession then SparkSession.builder.remote("sc://host:15002").getOrCreate() instead of .master(...) . Almost all DataFrame, SQL, Pandas-API-on-Spark, Structured Streaming, and Pandas UDF code migrates unchanged. The one class of code that does not migrate is anything touching SparkContext directly — sc.parallelize , sc.accumulator , sc.broadcast , df.rdd.map , custom partitioners. For the common 80% of PySpark code that is DataFrame + SQL + Pandas UDFs, the migration is two lines and the runtime cost is negligible. Spark Connect vs Spark Submit — when do I use which? Use Spark Connect when the client is interactive (notebook, IDE, dashboard), the client environment is slim (Lambda, Cloud Run, edge container), multiple clients share a server (notebook fleet, multi-language services), or the codebase is DataFrame/SQL/Pandas-UDF only. Use spark-submit when the client is a long-running batch JVM with no humans in the loop, the code uses RDD / accumulator / SparkContext APIs that Connect does not expose, or you have a strict batch-driver-lifecycle SLA (jobs must run to completion in a sealed driver process). The two shapes are complementary in 2026, not competitive: most teams end up running a Connect Server for interactive + DataFrame ETL and keeping spark-submit for RDD legacy and large batch jobs. The decision question that decides 95% of cases is "is the client interactive, or is it a batch JVM?" What APIs don't work in Spark Connect? The Connect protocol is DataFrame/SQL-only, so anything that requires direct SparkContext access is unavailable on the client. The concrete forbidden-API list: sc.parallelize (use spark.createDataFrame instead), sc.broadcast (use broadcast() from pyspark.sql.functions as a join hint), sc.accumulator (rewrite as a SQL aggregation), df.rdd.map / .flatMap / .reduceByKey (use mapInPandas , mapInArrow , or a SQL expression), df.rdd.partitionBy with custom partitioner (use df.repartition(N, "key") ), classic Java-RDD APIs ( JavaSparkContext , JavaRDD ), and some legacy Scala-class-of UDF overloads. UDF surface: Python UDFs and Pandas UDFs are first-class; Scala UDFs require uploading a JAR via AddArtifacts ; non-JVM clients (Go, Rust) cannot ship UDFs at all and are SQL-only for custom logic. Unity Catalog ACLs are enforced server-side using the auth token's identity.
