Why Tungsten exists — CPU is the new bottleneck spark tungsten was born because disk got fast, network got fast, and the JVM's object overhead became the cliff your job died on The one-sentence invariant: modern hardware made I/O cheap and the JVM expensive — Tungsten rewrites Spark's hot path so the CPU sees flat binary records, fused compiled operators, and a memory allocator that bypasses the garbage collector . Every other Tungsten consequence — UnsafeRow, whole-stage codegen, the unified memory pool, off-heap — falls out of that one premise. Once you internalise "CPU is the bottleneck, not disk," the entire tungsten engine interview surface reduces to "which Tungsten lever removes which CPU stall." The three problems Tungsten was designed to fix. JVM object overhead. A java.lang.Integer boxes a 4-byte int into roughly 16 bytes of object header plus 4 bytes of payload plus padding — 4x bloat for a single column. Multiply by a billion rows and you have an OOM that no amount of --executor-memory will fix. Tungsten replaces boxed-object rows with flat UnsafeRow byte buffers, slashing per-record memory by 4x–8x. Garbage collection pauses. A heap full of short-lived Row objects produces millions of GC roots per query; G1 / ZGC eventually catch up, but only after multi-second stop-the-world pauses on TB-scale shuffles. Tungsten's page-based allocator allocates large slabs (default 1MB pages) that are released as whole pages — the GC never sees the per-record allocations at all. Virtual dispatch and the volcano iterator. Pre-Tungsten Spark ran each operator as its own Java object with a virtual next() call; chaining filter → project → aggregate meant three virtual calls per record, three function-call frames, three branch-mispredicts. Tungsten's whole-stage codegen fuses those operators into a single compiled Java class with one loop — no virtual calls, predictable branches, hot-loop-friendly JIT optimization. The UnsafeRow layout starts with a null-bit-set sized to the next 8-byte boundary (one word covers all 3 nullable fields). At a billion rows, the boxed layout consumes roughly 97 GB; the UnsafeRow layout consumes roughly 33 GB — a 3x reduction in raw memory. Add GC overhead on the boxed side (typically 1.5x of the live set), and the gap widens to nearly 5x. The cluster-level implication: a 100-node executor pool with 1 GB of working memory per node holds 100 GB. Boxed rows do not fit; UnsafeRow does. Tungsten memory layout — UnsafeRow + binary format spark unsafe row is a flat byte buffer with a null-bit-set, fixed-length slots, and a variable-length region — and that single shape unlocks every other Tungsten win The mental model in one line: an UnsafeRow is a contiguous byte buffer laid out as [null-bit-set | fixed-length region | variable-length region] , where each field has a known offset, all fixed-length values are padded to 8 bytes, and variable-length values (strings, arrays, structs, maps) are stored in the trailing region with an 8-byte pointer in the fixed-length slot . Once you say "flat bytes with explicit offsets, no JVM objects," every spark binary format interview question becomes a deduction from that layout. The three regions of an UnsafeRow. Null-bit-set. The first 8 bytes (or N bytes rounded up to the next 8-byte word for > 64 fields) hold one bit per field. Bit i set → field i is null. Cheap to test ( & 1 << i ); cheap to clear. Fixed-length region. One 8-byte slot per field. For primitive types (int, long, float, double, boolean, byte), the value lives directly in the slot. For variable-length types, the slot holds a packed pointer (4-byte offset + 4-byte length) into the variable-length region. Variable-length region. The actual bytes of strings, byte arrays, structs, arrays, and maps. Packed back-to-back; the fixed-length slot stores offset (relative to row start) and length . Why fixed-length slots are always 8 bytes. An int is 4 bytes; a long is 8 bytes; storing them in 8-byte slots wastes 4 bytes per int. The trade is deterministic offset calculation : field i's offset is always 8 + nullBytes + 8 * i . Codegen can emit a single Platform.getInt(base, offset) call with no branching to look up the offset. Cache-line alignment is a side benefit: 8-byte slots align naturally to the 64-byte cache line on x86 and ARM. Sequential field reads are likely to hit the same line; predictable layout enables hardware prefetchers. The page-based off-heap allocator. Spark's TaskMemoryManager allocates memory in pages (default spark.buffer.pageSize=1 MB ). UnsafeRows are written into pages with their lengths prepended; rows can span pages but never start in the middle of one. Pages are tracked by an opaque 64-bit address: top 13 bits encode the page id, bottom 51 bits encode the in-page offset. Codegen and Tungsten internals use this address everywhere instead of raw pointers, so the allocator can migrate or free pages without breaking outstanding references — a soft form of compaction. On spark.memory.offHeap.enabled=true , pages are allocated via sun.misc.Unsafe.allocateMemory (or, on JDK 22+, the Foreign-Memory API). On false , pages come from a regular Java byte[] on the heap. The data layout is identical; only the backing store differs. Cache-friendliness — why the layout matters at the hardware level. A 64-byte L1 cache line on modern x86 holds 8 fixed-length slots. A filter that reads two ints from each row reads both from the same cache line — one memory access per row, not two. The variable-length region is laid out contiguously per row, so a string field follows the same row's int fields in memory. A typical projection (id, name) touches at most two cache lines. This is what people mean by "Tungsten is cache-aware." The layout was designed so that the access patterns Spark queries actually emit are friendly to the L1/L2 cache, branch predictor, and prefetcher. Off-heap allocation via sun.misc.Unsafe / Foreign-Memory API. sun.misc.Unsafe is an internal JDK API that grants raw pointer access — allocateMemory , putInt(address, value) , getLong(address) . Spark wraps it in a Platform helper that abstracts over both on-heap ( byte[] ) and off-heap ( long address ) backing stores. On JDK 22+, the Foreign-Memory API ( MemorySegment ) replaces Unsafe; Spark 4.x is migrating to it but maintains an Unsafe shim for older JDKs. The off-heap allocation does not count against -Xmx . It does count against spark.memory.offHeap.size and against the OS process's resident memory. Misconfiguring it is the #1 way to get killed by the OOM-killer instead of throwing a clean OOM. Whole-stage codegen — fused operators whole stage codegen fuses a chain of physical operators into a single compiled Java class — one loop, no virtual dispatch, JIT-optimisable The mental model in one line: Spark walks the physical plan, identifies the maximal chain of codegen-supported operators, and at runtime generates a single Java class that contains a fused while-loop calling inlined filter / project / aggregate logic on UnsafeRows from a single input source . Once you internalise "one Java class per stage, generated at runtime, then JIT-compiled," every spark codegen interview question becomes a deduction from "what fuses, what does not, and how do you read the explain output." The produce / consume protocol. Every codegen-supported physical operator implements two methods: produce(ctx) and consume(ctx, row) . produce emits the operator's own input loop (typically delegated to the upstream operator); consume emits the per-record processing logic (filter expression, projection, aggregate update). A WholeStageCodegen wrapper sits at the root of the fused subtree. It calls produce on its child; the child's produce recursively calls produce on its child, all the way down to the scan. The scan's produce emits the row-iteration loop; on each row, it calls consume on its parent; that parent's consume emits its own logic and then calls consume on its parent — bubbling up the chain. The result is a single Java method body that looks like while (input.hasNext()) { Row row = input.next(); /* filter */ if (cond) { /* project */ /* update hash map */ } } — no per-operator method calls, no per-record dispatch. The codegen-supported physical operators (the "fuses" list). Scans. FileSourceScanExec (Parquet, ORC, Avro, Delta with vectorized batch readers), InMemoryTableScanExec. Projections. ProjectExec (column selection, expression evaluation). Filters. FilterExec. Aggregates. HashAggregateExec, SortAggregateExec (the latter only when the input is already sorted). Joins. BroadcastHashJoinExec, ShuffledHashJoinExec, SortMergeJoinExec (the build side and probe side codegen separately). Generators. ExpandExec, GenerateExec (for explode, posexplode). Sort. SortExec. The codegen-break list (what stops fusion). Shuffle / Exchange. Any operator that requires a network shuffle breaks the codegen stage. The upstream side codegens up to the shuffle; the downstream side starts a new codegen stage from the read of the shuffle output. Python UDFs. A Python UDF cannot be inlined into Java source. The codegen stage breaks before the UDF; rows are materialized, shipped to a Python worker, results consumed back in a fresh stage. Use Pandas UDFs (vectorized) when possible to amortize the per-row crossing cost. Whole-stage-disabled operators. A handful of physical operators do not implement the codegen protocol (typically experimental or seldom-used ones). They appear in the plan as Subquery , Limit (in some forms), or custom extensions. Hugely complex expressions. Spark caps generated method size to keep the JIT happy — typically 8 KB of bytecode per method. Hugely complex expression trees that would generate more than that cause the codegen stage to fall back to the interpreted Volcano model. Look for org.codehaus.janino.JaninoRuntimeException: Code of method ... grows beyond 64 KB in the executor logs. Reading EXPLAIN CODEGEN . df.explain("codegen") (or EXPLAIN CODEGEN sql in Spark SQL) prints the generated Java source for every WholeStageCodegen node in the plan. Each WholeStageCodegen (N) block corresponds to one fused stage. The N is a sequence id used in stack traces and metrics. Stages are separated by codegen-breaking operators (Exchange, Python UDFs, etc.). The plan reads top-down; the first WholeStageCodegen block is the deepest in the tree (closest to the scan). The Java source includes the produce loop, the per-record consume body, and any HashMap-backed aggregate state. Reading it forces you to understand exactly what bytes the CPU is touching. When codegen is the win — and when it is not. Codegen wins big. Expression-heavy queries (lots of CAST, arithmetic, conditional logic), narrow scans with selective filters, hash aggregates with few keys, broadcast joins with selective probe sides. Typical speedup: 2-10x. Codegen wins modestly. Sort-merge joins (the bottleneck is the sort, not the merge), large hash aggregates with billions of keys (the HashMap dominates), high-fan-out generators. Typical speedup: 1.2-2x. Codegen does not win. Pipelines dominated by I/O (slow source, slow sink), pipelines dominated by serialization (Python UDFs everywhere, Kryo serialization on RDDs). Typical speedup: 1.0-1.2x — the bottleneck moved before codegen could help. Common interview probes on codegen. "What is whole-stage codegen and what does it generate?" — a single Java class with a fused while-loop per stage; compiled at runtime by Janino. "What breaks whole-stage codegen?" — shuffles, Python UDFs, the 8 KB method-size cap, certain experimental operators. "How do you inspect codegen output?" — df.explain('codegen') prints the generated Java; the Spark UI also exposes it per stage. "Why is codegen faster than the volcano iterator?" — eliminates per-record virtual dispatch and lets the JIT inline + vectorize the hot loop. Off-heap memory + execution memory model spark off-heap is a first-class memory pool — spark.memory.fraction dials the unified pool, off-heap toggles a parallel pool that sidesteps GC entirely The mental model in one line: Spark's executor memory is split into a unified on-heap pool (storage + execution sub-pools sharing one budget) and an optional off-heap pool sized by spark.memory.offHeap.size , where Tungsten's page-based allocator routes UnsafeRow pages . Once you internalise "one unified pool with a dial, plus an optional off-heap pool with its own switch," every spark memory management interview question reduces to "which pool, which knob." The executor memory anatomy. Reserved memory. ~300 MB hardcoded. Not user-tunable. Used for internal data structures. User memory. (1 - spark.memory.fraction) × (executorMemory - 300 MB) . Used for user objects (your closures, RDD-side data structures, broadcast variables not in the BlockManager). Default memory.fraction = 0.6 → user memory is 40% of usable heap. Unified memory. spark.memory.fraction × (executorMemory - 300 MB) . Shared by storage (cached blocks) and execution (Tungsten pages for shuffles, aggregations, sorts, joins). Default 60% of usable heap. Off-heap memory. Outside the JVM heap entirely. Sized by spark.memory.offHeap.size . Only used when spark.memory.offHeap.enabled=true . Storage vs execution — the unified-pool semantics. Storage memory holds cached RDD/DataFrame blocks (the result of df.cache() or df.persist() ). It is configured by spark.memory.storageFraction (default 0.5 → half of the unified pool). Execution memory holds Tungsten pages for shuffle buffers, hash-map state for aggregates, sort buffers, broadcast-join hash tables. Borrowing. If execution needs memory and storage has free space (or evictable blocks), execution borrows. Storage can evict cached blocks to free space for execution but can never reclaim memory from execution mid-task — execution memory is sacred to the running task. This is the unified memory manager (default since Spark 1.6). The older static memory manager had separate pools that could not borrow — fixed sizes, common OOMs on shuffle-heavy jobs. The off-heap toggle. spark.memory.offHeap.enabled=true + spark.memory.offHeap.size=<bytes> enables off-heap pages. Off-heap memory is allocated via sun.misc.Unsafe.allocateMemory (or Foreign-Memory API on JDK 22+). It is not counted against -Xmx . It is counted against the OS process's resident memory. Off-heap pages contain UnsafeRows, hash-map buckets, and sort buffers — the same Tungsten data structures, in a different backing store. The trade-off. Off-heap memory is invisible to the GC, so it eliminates GC pauses on the working set. The cost is operational: misconfigure the size and the OS OOM-killer terminates the executor (with no clean Spark-side error message), and native-memory diagnostics are harder than JVM-heap diagnostics. When off-heap pays back. Large working sets. > 8 GB of state per executor where GC pauses become 100+ ms. Off-heap moves the working set out of the GC's view. Latency-sensitive batch jobs. When a 200ms GC pause per executor blows your SLA. Off-heap pauses are predictable — there are none on the off-heap data, only on the small on-heap residual (closures, broadcast vars). Multi-TB shuffles. When the sort-buffer working set is enormous and the on-heap budget can't hold it. Off-heap is the only way to fit. With on-heap, the alternative is spilling to disk — much slower. When off-heap is a trap. Small jobs. < 4 GB working set. The GC handles this fine; off-heap adds operational complexity for no measurable win. Containerized clusters with tight memory limits. YARN / K8s containers enforce a hard cap on total memory. Off-heap counts against this cap; if the executor heap + off-heap + JVM overhead exceeds the container limit, the container is killed without warning. Always size containers as executorMemory + memoryOverhead + offHeapSize + headroom . Teams without native-memory profiling tooling. Debugging a native OOM is harder than a heap OOM. Without Async Profiler or NMT (Native Memory Tracking), an off-heap leak is opaque. The page-based allocator — same shape, two backing stores. On-heap pages: backed by new byte[(int) pageSize] . The page reference is a Java byte[] ; the GC tracks it. Page address packing uses the byte array's identity-hash-based id. Off-heap pages: backed by Unsafe.allocateMemory(pageSize) . The page is a raw long address; the GC does not see it. The allocator manually frees pages on task completion. UnsafeRow reads use the same Platform.getLong(baseObj, baseOff) API for both. baseObj is the byte[] (on-heap) or null (off-heap); baseOff is the array-relative offset or the absolute address. The codegen-generated code does not branch on the backing store. Common interview probes on the memory model. "What is the difference between storage and execution memory?" — storage caches blocks; execution holds Tungsten pages. They share a unified pool and borrow from each other. "When do you enable off-heap?" — large working sets where GC pauses dominate, with operational tooling to debug native-memory issues. "Why is execution memory sacred to the running task?" — because evicting a Tungsten page mid-task would crash the task. Storage blocks can be re-read from disk; execution pages cannot. "How do you size off-heap?" — measure the largest live state across recent runs (Spark UI → Executors → Storage and Shuffle), add ~30% headroom, double-check it fits in the container limit. Unified is shared by storage and execution; user holds your closures, accumulators, and other JVM objects not in the cache/execution pool. Inside the unified pool, storageFraction=0.5 sets the initial storage budget at half. This is a starting point — execution can borrow from storage (evicting cached blocks), and once execution holds memory, storage cannot reclaim it mid-task. With offHeap.enabled=true and offHeap.size=8g , a parallel off-heap pool is created. Its internal split mirrors the on-heap one: half storage, half execution by default. The total memory budget your YARN / K8s container needs to provision is executor + overhead + offHeap . With a 10% overhead default, the 16 GB heap + 8 GB off-heap container needs ~25.6 GB. Forgetting the off-heap line item in container sizing is the #1 cause of unexplained executor terminations. Output (memory budget). Tuning Tungsten in production Senior engineers tune Tungsten with 4 levers in order — EXPLAIN CODEGEN , off-heap toggle, vectorized Parquet reader, Async Profiler — never with cargo-cult configs The mental model in one line: production Tungsten tuning is a 4-step diagnosis — read EXPLAIN CODEGEN to find codegen breaks, toggle off-heap only when GC dominates, ensure vectorized columnar readers are on the hot scans, then validate with Async Profiler / JFR flame graphs . Once you internalise "diagnose then tune, in that order," every spark cpu efficiency interview question becomes a deduction from the diagnostic ladder. Lever 1 — EXPLAIN CODEGEN for codegen breaks. Every slow job starts with df.explain("codegen") or df.explain("formatted") . Look for the *(N) prefixes and identify boundaries between codegen stages. Common findings: a BatchEvalPython node in the middle of a hot pipeline (Python UDF break), an Exchange that you didn't expect (extra shuffle from an orderBy or repartition ), a Subquery node that escaped optimization. The fix is usually a code-level change: rewrite the Python UDF as a Pandas UDF or built-in SQL expression, eliminate the unnecessary orderBy , pre-bucket the source to skip the shuffle. Lever 2 — off-heap toggle (only when GC is the bottleneck). Check GC time in the Spark UI's Executors tab. If GC time / task time > 5% , GC is dominating. Enable off-heap with spark.memory.offHeap.enabled=true and spark.memory.offHeap.size=<bytes> . Re-size the container. Re-measure GC time — expect it to drop to ~1% of task time. If it doesn't drop, the bottleneck wasn't GC; revert. Common mistake: enabling off-heap with no container resize → executor OOM-killed by YARN/K8s with no Spark-side error. Lever 3 — vectorized columnar readers. spark.sql.parquet.enableVectorizedReader=true (default in modern Spark). Reads Parquet column chunks as batches of UnsafeRows; ~2-3x faster than the row-by-row reader. spark.sql.orc.enableVectorizedReader=true (default for ORC). For Delta Lake, vectorized readers are enabled by default with the Delta Spark connector. Verify by checking the physical plan: a vectorized scan appears as ColumnarToRow + FileScan and is wrapped in WholeStageCodegen . A row-based scan appears as FileScan directly (no ColumnarToRow ) and is sometimes outside codegen. Lever 4 — Async Profiler + Java Flight Recorder. Async Profiler attaches to a running executor JVM, samples the CPU at 100 Hz, and emits a flame graph showing where the time is actually spent. Frequently asked questions What is Spark Tungsten? Spark Tungsten is a layered execution engine inside Apache Spark designed to make the CPU the dominant cost driver instead of memory and dispatch overhead. It has three braided components: a binary memory layout ( spark unsafe row — flat byte buffers with explicit offsets, replacing boxed JVM rows), whole-stage codegen ( spark codegen that fuses chained physical operators into a single compiled Java class with one hot loop), and a page-based memory manager (a unified storage+execution pool with optional spark off-heap allocation via sun.misc.Unsafe or the Foreign-Memory API). Together they reduce per-record memory 3-5x, eliminate per-record virtual dispatch, and let the JIT optimize the fused loop as a single hot path. The tungsten engine was introduced in Spark 1.5, became default-on in Spark 2.0, and underpins every modern Spark workload — Photon, Connect, Structured Streaming all build on the same foundation. Why is whole-stage codegen faster? Whole-stage codegen fuses a chain of physical operators (filter, project, aggregate) into a single compiled Java class with one inlined while (input.hasNext()) loop. In the pre-Tungsten volcano model, each operator was a separate object with a virtual next() call, so a 3-operator chain meant 3 virtual dispatches, 3 stack frames, and 3 chances for branch mispredict per record. With codegen, all three operators' logic lives in one method body — the JIT inlines, applies bounds-check elimination, and on supported JVMs auto-vectorizes the loop. Typical speedups are 2-10x on expression-heavy queries. The cost is a one-time compilation step (~100-500 ms per stage) that batch jobs amortize trivially. Codegen breaks at shuffle boundaries, Python UDFs, and any operator that does not implement the produce/consume protocol — and recognising those breaks is the senior-DE signal whole stage codegen interviewers probe for. When should I enable off-heap memory? Enable spark.memory.offHeap.enabled=true when three conditions hold: the working set is large (> 8 GB per executor), GC pauses dominate the latency tail ( GC time / task time > 5% ), and your YARN/K8s container limit can accommodate the additional offHeap.size line item. The mechanism is straightforward — Tungsten's page allocator routes UnsafeRow pages to sun.misc.Unsafe.allocateMemory (or the Foreign-Memory API on JDK 22+) instead of byte[] on the heap, so the GC never sees the working set. Streaming jobs with multi-GB state, multi-TB shuffles, and long-running batch jobs with massive aggregates are the classic wins. Skip off-heap for small jobs (< 4 GB working set), tightly containerized clusters where the off-heap line item would exceed the container limit, or teams without native-memory profiling tooling (Async Profiler, NMT). The senior answer is "off-heap is a power tool, not a default." What breaks whole-stage codegen? Four things break spark codegen : shuffles (every Exchange operator ends one codegen stage and starts a new one — the network shuffle is a natural fence), Python UDFs (scalar Python UDFs route each row through a Python worker over a socket; Pandas UDFs batch via Arrow but still break the Java codegen fusion), the 64 KB generated-method cap (extremely wide schemas or complex expression trees produce Java methods that exceed Janino's compile limit and fall back to volcano-style interpretation — look for JaninoRuntimeException: Code of method ... grows beyond 64 KB in executor logs), and experimental or unimplemented physical operators (a handful of plan nodes do not implement the produce/consume protocol). Diagnose with df.explain("codegen") or df.explain("formatted") — operators with the same *(N) prefix are in the same fused stage; boundaries between Ns are breaks. The fix is usually a query rewrite (Pandas UDF instead of scalar, eliminate unnecessary orderBy , broadcast-join instead of shuffle-join when possible), occasionally a config tweak ( spark.sql.codegen.methodSplitThreshold ), rarely a Spark-version bug. UnsafeRow vs Row — when does the difference matter? UnsafeRow is Tungsten's flat byte buffer — [null-bit-set | fixed-length slots (8 bytes each) | variable-length region] — and is what every DataFrame / Dataset operation produces and consumes through Catalyst. A boxed Row is a JVM object with per-field references, ~3-5x the memory footprint, and a target for the garbage collector. The difference matters whenever you (a) hold rows in memory at scale (> 100M rows), (b) hit GC pauses on a query that should be CPU-bound, or (c) drop into the RDD API via .rdd or a .map(row => …) Dataset lambda — at that point you exit the Tungsten path and revert to boxed-row semantics in your lambda, paying the 3-5x memory penalty. The spark binary format is also what makes whole-stage codegen, off-heap allocation, and vectorized columnar readers all work as a coherent system — every layer assumes UnsafeRow's deterministic offsets. The senior signal is "I keep my hot paths inside the DataFrame / SQL API to preserve UnsafeRow end-to-end." Spark memory model — storage vs execution pools Spark's executor memory has four regions: a hardcoded 300 MB reserved block, a user pool for closures and accumulators ( (1 - memory.fraction) × usable , default 40%), and a unified pool ( memory.fraction × usable , default 60%) shared by storage (cached blocks) and execution (Tungsten pages for shuffles, aggregates, sort buffers). Inside the unified pool, storageFraction=0.5 sets the initial split, but the pool self-balances: execution can evict cached storage blocks to grow, while storage cannot reclaim memory from a running task (execution memory is sacrosanct mid-task). Enabling spark.memory.offHeap.enabled=true adds a parallel off-heap pool sized by offHeap.size , with the same internal storage+execution split. The page-based allocator routes UnsafeRow pages to whichever pool is active. spark memory management interviewers expect you to walk through all four regions, name storageFraction and memory.fraction , and explain the borrow rules — that is the senior-DE depth signal.
