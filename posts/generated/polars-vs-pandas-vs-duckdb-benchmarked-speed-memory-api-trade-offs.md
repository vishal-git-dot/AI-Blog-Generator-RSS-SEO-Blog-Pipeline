Worked example — same groupby, three engines, three timings Detailed explanation. The classic dataframe Hello World — sum a revenue column by country from a 10GB Parquet file — looks deceptively similar in all three engines. The methods read almost identically. The runtime is wildly different: pandas walks one thread over Python objects, Polars dispatches across all cores over Arrow buffers, and DuckDB compiles a vectorised pipeline and streams Arrow chunks. Question. Compute total_revenue per country over a 10GB Parquet file in pandas (eager), Polars (lazy + collect), and DuckDB (SQL). Highlight the runtime difference (wall clock and peak RAM), and explain why the gap is structural, not implementation detail. Input. Column Type Cardinality country string ~250 customer_id int64 50M revenue float64 continuous Code. # pandas — eager, single-threaded, Python-object overhead import pandas as pd df = pd . read_parquet ( " sales.parquet " ) out = df . groupby ( " country " , as_index = False )[ " revenue " ]. sum () out . to_parquet ( " by_country_pandas.parquet " ) # wall: ~90s, peak RAM: ~28GB (object dtype overhead + intermediate) # Polars — lazy, multi-threaded, Arrow native import polars as pl out = ( pl . scan_parquet ( " sales.parquet " ) . group_by ( " country " ) . agg ( pl . col ( " revenue " ). sum ()) . collect ( streaming = True ) ) out . write_parquet ( " by_country_polars.parquet " ) # wall: ~6s, peak RAM: ~4GB (Arrow buffers + streaming) # DuckDB — SQL, columnar, cost-based optimiser import duckdb con = duckdb . connect () con . execute ( """ COPY ( SELECT country, SUM(revenue) AS total_revenue FROM ' sales.parquet ' GROUP BY country ) TO ' by_country_duckdb.parquet ' (FORMAT PARQUET) """ ) # wall: ~4s, peak RAM: ~2.5GB (vectorised pipeline + spill) Step-by-step explanation. Pandas materialises the entire Parquet file into a DataFrame with object-dtype strings for country . Every value carries a Python-object header (~50 bytes per string) so the in-memory size is several times the on-disk size. The groupby(...).sum() call builds a Python-level hash map keyed by Python string objects. Hashing a Python string and incrementing a counter is a fast operation, but doing it 50M times in one thread takes time. Polars's scan_parquet returns a LazyFrame — a description of the work, not the data. The group_by and agg calls build a query plan. Only collect(streaming=True) triggers execution. The Polars optimiser reorders operations, prunes columns (only country and revenue are read off disk — the other columns are skipped at the Parquet level), and dispatches the work across all CPU cores. Each thread hashes a chunk of Arrow buffers. DuckDB parses the SQL, builds a logical plan, runs the cost-based optimiser ( country cardinality vs row count chooses hash-aggregate over sort-aggregate), and compiles a vectorised pipeline. Arrow blocks of ~2048 rows flow through the operators with no per-row Python overhead. The three timings — 90s, 6s, 4s — are not measurement noise. They are the result of three execution models on the same workload. The pandas → Polars gap is dominated by threading + Arrow ; the Polars → DuckDB gap is dominated by plan optimisation + vectorised pipeline fusion . Output (wall time + peak RAM benchmark). Execution model — eager vs lazy vs SQL polars vs pandas and polars vs duckdb are an execution-model story — eager Python, lazy Rust, and cost-based SQL planner respectively The mental model in one line: pandas runs each statement eagerly the moment Python evaluates it, Polars builds a polars lazy frame plan and only executes on .collect() , and DuckDB parses your SQL into a logical plan, runs a cost-based optimiser, and compiles a vectorised pipeline — and that single difference dictates which engine wins which workload . Once you say "eager Python vs lazy Rust vs SQL planner," every polars benchmark interview question becomes a deduction from those three execution models. The pandas execution model. Eager — every statement runs immediately. df.assign(x=...) allocates a new DataFrame; df["x"] = ... mutates in place. There is no plan, no deferred execution. Single-threaded — the GIL bites. Most pandas operations run in one Python thread. Some NumPy / vectorised paths drop the GIL internally, but the user-visible model is "one core." NumPy-backed by default; PyArrow-backed since 2.0. The default object -dtype path stores Python objects (slow, memory-heavy). The opt-in PyArrow backend ( dtype_backend="pyarrow" ) stores Arrow buffers — much faster, much less memory, but you have to ask for it. The mental model is "Series + Index." Every pandas operation is a Series-on-Series operation; the index aligns the two sides. Misaligned indices silently change semantics. The Polars execution model. Lazy — the plan is the program. pl.scan_parquet(...).filter(...).group_by(...) builds a LazyFrame description; .collect() triggers execution. Multi-threaded — Rust core, no GIL. Polars dispatches work across all CPU cores. The Python layer is a thin wrapper around a Rust engine that does not touch the GIL during execution. Arrow-native — zero-copy with the ecosystem. A Polars DataFrame is an Arrow table. You can hand it to DuckDB, PyArrow, or pandas 2.x without copying. The plan is optimised before execution. Predicate pushdown, projection pushdown, common-subexpression elimination, join reordering, slice pushdown — Polars applies all of them on the lazy plan. The DuckDB execution model. SQL-first — parse, plan, optimise, execute. A SQL query goes through a parser → logical plan → optimiser → physical plan → vectorised executor. Cost-based optimiser. DuckDB collects statistics about cardinalities and column distributions and uses them to reorder joins (smallest-table-first), pick the right join algorithm (hash vs merge vs nested-loop), and choose hash-aggregate vs sort-aggregate. Vectorised execution over Arrow chunks. DuckDB's runtime processes ~2048-row "chunks" through a pipeline of operators. Each operator works on whole chunks at once — no per-row Python overhead. In-process + zero-copy to dataframes. DuckDB runs inside your Python process. It can query a pandas DataFrame or a Polars DataFrame as a virtual table with no data movement. The 2026 reality — what changed since 2022. Polars 1.x stabilised the streaming engine — large queries spill intermediates to disk so the engine no longer requires the working set to fit in RAM. The streaming engine is on by default in Python 3.13+. pandas 2.x added the PyArrow backend and the dtype_backend="pyarrow" toggle. String columns went from "object Python strings" to "Arrow strings"; performance and memory both improved 2-4x. DuckDB 1.x stabilised the on-disk format (Parquet, Iceberg, native DuckDB) and introduced async I/O, parallel CSV / Parquet readers, and a much-improved out-of-core groupby with disk spill. Apache Arrow is now the lingua franca. All three engines use Arrow buffers internally; zero-copy interop is free. Common interview probes on execution model. Every operation runs eagerly; the engine cannot rearrange your code. "What is the vectorised execution model in DuckDB?" — operators process columnar chunks (~2048 rows) at a time; no per-row Python overhead. Step Operation 1 scan_parquet('sales.parquet') 2 filter(pl.col('country') == 'US') 3 select(['country', 'month', 'revenue']) 4 group_by('month').agg(pl.col('revenue').sum()) Code. import polars as pl plan = ( pl . scan_parquet ( " sales.parquet " ) . filter ( pl . col ( " country " ) == " US " ) . select ([ " country " , " month " , " revenue " ]) . group_by ( " month " ) . agg ( pl . col ( " revenue " ). sum ()) ) print ( plan . explain ( optimized = False )) # raw plan print ( plan . explain ( optimized = True )) # what actually runs result = plan . collect () Step-by-step explanation. The unoptimised plan executes top-down: scan all columns of the Parquet file, filter, select 3 columns, group, aggregate. This would read every byte of the file. The optimiser walks the plan and applies projection pushdown: the final select plus the agg only use month and revenue , plus the filter needs country . So only those three columns need to be read from Parquet. Predicate pushdown: the filter country='US' is moved into the Parquet scan. Parquet's row-group statistics let the reader skip row groups where no country='US' row exists. With both pushdowns applied, the optimised plan reads ~10% of the rows × 15% of the columns ≈ 1.5% of the original file's bytes from disk. The aggregation runs in parallel across the partitions. The result is collected back to a single DataFrame. Output (plan optimisation effect). If projection or predicate pushdown is not happening, you probably wrote the query in a way that defeats the optimiser (e.g. an opaque Python UDF that the planner cannot analyse). The SQL writes the joins as orders × customers × countries . Internally, DuckDB picks a HASH_JOIN operator for both joins (smaller side fits in memory). For the countries join, the build side is 250 rows — trivial. For the customers × orders join, the build side is 1M rows — fits comfortably. Memory profile — Arrow, streaming, spill pandas vs duckdb on memory is a 5-10x story — arrow columnar storage, Polars streaming, and DuckDB external spill change what runs on a laptop The mental model in one line: pandas stores Python objects in row-friendly NumPy arrays (3-5x memory blow-up over disk), Polars stores Arrow columnar buffers (close to disk size, can stream out-of-core), and DuckDB streams Arrow chunks through a vectorised pipeline that spills to disk when the working set exceeds memory — and that difference is what turns "this query OOMs my laptop" into "this query runs on my laptop" . Once you say "object dtype vs Arrow vs streaming vs spill," every polars vs duckdb memory question becomes a deduction from those four memory models. The pandas memory profile. Object dtype is the killer. A column of strings stored as object is a NumPy array of Python str objects. Each str carries ~50 bytes of Python header plus the string bytes. A 1GB string column on disk often becomes 5GB in RAM. NumPy backend — copy-on-write since 2.0. Pandas 2.0 introduced pd.options.mode.copy_on_write = True , which avoids unnecessary copies on chained operations. Recommended for any 2.x deployment. PyArrow backend — opt-in. dtype_backend="pyarrow" switches string columns to Arrow strings, integer columns to Arrow integers, etc. Memory drops 2-4x; speed rises 2-3x. Intermediate materialisation. Every step of a pandas pipeline produces a full intermediate DataFrame. df.assign(x=...).filter(...).groupby(...).sum() materialises three intermediates. The Polars memory profile. Arrow buffers everywhere. Every column is an Arrow buffer — primitive types take exactly their bit width, strings are stored as a single contiguous bytes buffer + offset array. No per-element Python overhead. Streaming engine — out-of-core. Polars's streaming engine runs the query in chunks. Intermediate state lives in bounded buffers; data that exceeds memory spills to a sink and is processed in batches. No intermediates in lazy plans. The optimiser fuses adjacent operations into pipelines that pass Arrow chunks operator-to-operator without materialising the full intermediate. Categorical and dictionary encodings. Polars's categorical (and Arrow's dictionary) encoding stores low-cardinality string columns as (int, dictionary) pairs — often 10x smaller than naive strings. The DuckDB memory profile. Vectorised execution, not materialised execution. DuckDB does not materialise full intermediate tables. Operators process Arrow chunks of ~2048 rows; only the chunk and small per-operator state are in memory. External spill — disk fallback for big aggregations. When a hash-aggregate's hash table exceeds the memory budget, DuckDB spills partitions to disk and processes them in two passes. Memory budget is a knob. PRAGMA memory_limit='8GB' tells DuckDB to use at most 8GB; above that, it spills. Zero-copy with Arrow. A Polars DataFrame registered with con.register(...) shares its Arrow buffers with DuckDB — no double allocation. The 2026 reality — what changed since 2022. Pandas 2.x PyArrow backend narrowed the gap. String-heavy workloads that were 5x larger in pandas vs Polars are now 1.5-2x larger. Polars streaming engine stabilised. Out-of-core joins and group-bys that were beta in 2023 are now the default in Python 3.13+. DuckDB external spill matured. Multi-TB groupby on a 32GB laptop is now routine; the spill mechanism is transparent. Arrow IPC (Inter-Process Communication) lets you hand a DataFrame from one process to another with no serialisation — useful for multi-process pipelines. Common interview probes on memory. "Why does my pandas DataFrame use 5x its on-disk size?" — object-dtype strings; switch to PyArrow backend or categorical. "What is Polars's streaming engine?" — an out-of-core execution mode where the engine processes data in chunks and spills intermediates to disk if needed. "What does DuckDB do when a join exceeds memory?" — partitioned hash join with external spill: build side partitioned to disk, probe side iterated against each partition. "Is Arrow the same as Parquet?" — no. Arrow is an in-memory columnar format; Parquet is an on-disk columnar format. Arrow ↔ Parquet conversion is the default for reading / writing. API ergonomics — index vs expressions vs SQL polars dataframe expressions, pandas index semantics, and DuckDB SQL are three different ways to write the same query — pick the one your team writes fastest The mental model in one line: pandas's API is built around a Series + Index abstraction (every column is a Series, every operation aligns on the index), Polars's API is built around expressions on columns (every operation is a pl.col(...).<expr> over the whole frame), and DuckDB's API is SQL (parsed, planned, executed) — and the right API for a team is the one its engineers write fastest under pressure . Once you say "index vs expression vs SQL," every polars dataframe API question becomes a comparison of those three mental models. Pandas — Series + Index. Every column is a Series . A Series is a 1D labelled array with an Index. Operations between two Series align on the Index — that is the pandas idiom. The Index is a first-class object. Reset, set, multi-index, hierarchical — the index is what makes pandas powerful for lookups and what makes it confusing for newcomers. Inplace vs copy is its own world. df["x"] = ... mutates; df.assign(x=...) returns a new DataFrame; df.loc[..., "x"] = ... mutates with index alignment. The 2.x copy_on_write mode standardises this — copy semantics by default, explicit mutation when needed. Method chaining is possible but fragile. Pipelines are usually broken across df1 = ... , df2 = df1.assign(...) lines because chaining preserves intermediate state. Polars — expressions + chains. Every operation is a pl.col(...).<expr> . Expressions are first-class; you can save them, reuse them, build them programmatically. with_columns adds columns; select projects; group_by + agg aggregates. Three primary verbs, plus filter , join , sort , pivot . No index. Polars deliberately omits the pandas Index. Joining is always explicit ( how="inner" , on="key" ). Aligning two columns is always explicit. Method chaining is the natural idiom. A whole pipeline reads top-to-bottom as one . chain. Easy to read once you train your eye. DuckDB — SQL. The query is a SQL string. Subqueries, CTEs, window functions, set operations — all standard SQL. Relational API exists too. con.table("orders").join(...).aggregate(...) is a fluent API that maps to SQL internally. Most users stay on raw SQL. Result types. con.execute(...).df() returns a pandas DataFrame; .pl() returns a Polars DataFrame; .arrow() returns a PyArrow Table; .fetchall() returns Python tuples. SQL is portable. A DuckDB SQL query is mostly the same query you would run on Snowflake / BigQuery / Postgres. The translation cost is small. Interop — Arrow ties them together. Pandas → Polars. pl.from_pandas(df) — zero-copy for Arrow-backed pandas; otherwise a fast conversion. Polars → pandas. df.to_pandas(use_pyarrow_extension_array=True) — zero-copy with the PyArrow extension array. Polars → DuckDB. con.register("name", polars_df) — zero-copy; the DataFrame is a virtual table. DuckDB → Polars. con.execute(...).pl() — converts the result to a Polars DataFrame. Pandas → DuckDB. con.register("name", pandas_df) — zero-copy for Arrow-backed pandas; otherwise PyArrow converts under the hood. Common interview probes on API. "What is the difference between df.assign and df["x"] = ... in pandas?" — assign returns a new DataFrame; df["x"] = ... mutates in place. "What is an expression in Polars?" — a first-class object representing a computation on a column; can be saved, reused, composed. "Why does Polars not have an index?" — Polars treats the index as an implicit alignment that creates bugs in pandas. Explicit joins are clearer. "How do I use SQL on a Polars DataFrame?" — duckdb.query("SELECT ... FROM polars_df") works directly; polars_df.sql("...") also exists. Yes — typically 5-10x faster on the same workload, sometimes 20x+ on large groupby / join pipelines. A polars lazy frame is a description of a query — operations, expressions, sources — without any data.
