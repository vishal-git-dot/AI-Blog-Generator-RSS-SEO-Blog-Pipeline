# Spark Catalyst Optimizer & Adaptive Query Execution (AQE): Plan Internals for 2026

## Overview

spark catalyst optimizer is the single most leveraged piece of code in the modern data platform — every Spark SQL query, every DataFrame call, every Delta / Iceberg / Parquet read passes through it before a single byte moves — and the part most engineers never look at until production is on fire. Catalyst is the rule-based + cost-based query planner; spark aqe is the runtime feedback loop that sits on top, rewriting plans between stages when the optimizer's compile-time guesses turn out wrong. Together they are the reason df.join(other).groupBy(...).agg(...) doesn't behave like the naive Spark you remember from 2017. Why Catalyst + AQE define Spark's 2026 query story Catalyst is the compile-time optimizer, AQE is the runtime feedback loop — every Spark performance conversation starts at one of those two layers The one-sentence invariant: Catalyst rewrites your DataFrame / SQL into an optimised physical plan before execution; AQE re-rewrites that plan during execution as each shuffle stage reports real row counts back to the driver . Every "Spark is slow" debugging session that doesn't start at one of those two layers is doomed to chase symptoms. The configs, the broadcast hints, the partition-count knobs — all of them are levers on Catalyst and AQE . If you don't know what the optimizer is doing, you are tuning blindfolded. The four axes interviewers actually probe. Rules. Catalyst ships dozens of rule families — column pruning, predicate pushdown, constant folding, projection collapse, boolean expression simplification, join reorder. Each rule is a Rule[LogicalPlan] subclass that runs in batches under a RuleExecutor . Senior engineers can name three to five families unprompted. Plan. Every query produces four plans in order: unresolved logical plan , analyzed plan (schema bound), optimized logical plan (rules applied), physical plan ( SparkPlan operators). df.explain("extended") prints all four; df.explain("formatted") (Spark 3.0+) prints a compact tree with a side table of operator details. Statistics. Cost-based optimization (CBO) needs row counts and column histograms. Without ANALYZE TABLE ... COMPUTE STATISTICS , Catalyst falls back to size-only estimates and often picks the wrong join order. CBO is off by default; senior engineers know when to flip it on. Runtime feedback. AQE is the loop that closes the planner blind-spot. It cannot predict shuffle output sizes; only the actual run can. AQE samples the shuffle map output, coalesces tiny partitions, splits skewed ones, and swaps sort-merge joins for broadcast hash joins when one side comes in smaller than the broadcast threshold. Why "Spark is slow" answers usually miss the optimizer. The bottleneck is almost never the language. It is one of: (a) a misplanned shuffle, (b) a skewed key, (c) a broadcast that didn't fire, (d) a partition-prune that didn't fire, (e) whole stage codegen blocked by a Python UDF. All five are plan-level concerns, visible only in EXPLAIN . "Add more executors" rarely helps a misplanned plan. A 5x cluster with a skewed plan still has one core doing 80% of the work. "Cache the DataFrame" doesn't help if the plan above the cache is wrong; you cached the wrong intermediate. The first move on any "Spark is slow" ticket: open the Spark UI's SQL tab , find the slow query, read its plan, find the stage with the long-tail task. Only then start touching configs. What changed in 2026 — Spark 3.5 / 4.0 era. AQE on by default since Spark 3.2 ( spark.sql.adaptive.enabled=true ). New jobs inherit the feedback loop without opt-in. Storage-Partitioned Joins (SPJ) — a Catalyst rule that recognises when two tables share a partition layout and skips the shuffle entirely. Big win on Iceberg + Delta partitioned tables. Spark 4.0 ANSI SQL by default — Catalyst now treats overflow / division-by-zero as errors unless you opt out, surfacing planning-level type coercions earlier. Photon-style codegen in Databricks Runtime — vectorised executors that piggyback on Catalyst's physical plan; outside Databricks, the same physical plan still runs through Spark's whole-stage codegen. CBO + AQE are complementary , not redundant. CBO picks the initial plan; AQE corrects it at runtime. Big tables benefit from both. What interviewers listen for. Do you say "Catalyst at compile time, AQE at runtime" in the first sentence? — senior signal. Do you mention "four plans: unresolved, analyzed, optimized, physical" unprompted? — required answer. Do you bring up "AQE coalesces, splits, and swaps to broadcast" as the three big features? — senior signal. Do you push back on "AQE replaces tuning" with "AQE corrects what the planner couldn't predict; you still set good initial partition counts"? — senior signal. The customers table is 80MB; the orders table is 4TB partitioned by date. What does Catalyst produce in 2018 Spark vs 2026 Spark with AQE on?" The DataFrame code looks identical. The physical plan, the shuffle, and the runtime behaviour are wildly different. Question. Write a customers.join(orders) aggregation, then list the four planning stages each version of Spark produces, the runtime rewrites AQE can apply, and the resulting stage shape. Input. Table Size Partitioning customers 80 MB none orders 4 TB by order_date (3650 partitions) target one row per (customer_id, month) aggregation Code. from pyspark.sql import functions as F monthly = ( customers . alias ( " c " ) . join ( orders . alias ( " o " ), F . col ( " c.customer_id " ) == F . col ( " o.customer_id " )) . where ( F . col ( " o.order_date " ) >= " 2025-01-01 " ) . groupBy ( " c.customer_id " , F . date_trunc ( " month " , " o.order_date " ). alias ( " month " )) . agg ( F . sum ( " o.amount " ). alias ( " total " )) ) monthly . explain ( " formatted " ) monthly . write . mode ( " overwrite " ). parquet ( " s3://lake/curated/monthly " ) Step-by-step explanation. The DataFrame is not executed yet — Spark builds an unresolved logical plan from the lazy operators. Column names are strings; types are unknown. The analyzer resolves names against the Spark catalog ( customers , orders ), binds types, and emits an analyzed plan . Any missing column would fail here, not at run time. The optimizer applies rule batches. where("order_date >= '2025-01-01'") is pushed through the join into the orders scan ( predicate pushdown ); only customer_id , order_date , amount are read ( column pruning ); the date_trunc("month", ...) is folded into a single Catalyst expression. The result is the optimized logical plan . The planner maps logical operators to physical ones via the Strategy API. Without statistics, the planner usually picks SortMergeJoin . With CBO and accurate stats, the 80MB customers side is small enough to trigger BroadcastHashJoin . AQE-era (Spark 3.2+) runtime feedback: even if the planner missed the broadcast, AQE may observe that the customers shuffle output is < 10MB and rewrite the plan mid-query to a broadcast hash join. AQE also coalesces the post-shuffle partitions on the orders side (from 200 down to ~40 if the data is sparse) and splits any heavily-skewed customer. Output. Catalyst handles the rule-based wins (pushdown, pruning, folding). AQE handles the runtime wins (broadcast swap, coalesce, skew split). Squashing them into "the optimizer" is the most common misconception; senior engineers name all four. Rule batches in Catalyst — rules run in batches under a RuleExecutor . A batch is "run to fixed point or N iterations." Knowing this explains why some optimisations need multiple passes (predicate pushdown enables projection collapse enables more pushdown). CBO is off by default — most teams discover this after months. Without ANALYZE TABLE , Catalyst is in size-only mode. CBO is a one-line flag; the stats are the work. AQE wraps the physical plan — it does not replace it. The compile-time plan is the starting point; AQE rewrites downstream stages once a shuffle's actual output is observable. Codegen is per WSCG block — Spark fuses adjacent operators with the same shuffle boundary into a single Java method. Python UDFs break the block; Pandas UDFs (Arrow) keep it intact. Cost — planning is O(operators) and happens once per query. Codegen is O(stages) and is cached. AQE rewrite is O(stages × rewrite cost) and pays for itself when the rewrite avoids a full shuffle. Catalyst — analyzer to optimizer to planner spark logical plan becomes a spark physical plan through four stages — rules in the middle do the heavy lifting The mental model in one line: Catalyst is a tree-rewriting compiler — your DataFrame call is a tree, every optimisation is a pattern-match-and-rewrite rule, every plan is a tree, and the rules run in ordered batches until the tree stabilises . Once you say "Catalyst rewrites trees," every spark optimizer rules interview question becomes a deduction from "which rule pattern-matches this fragment and what does it rewrite to?" The four plan trees, in order. Unresolved logical plan. Built directly from the user's DataFrame or SQL. Operators are Project , Filter , Join , Aggregate , Relation . Attribute references are unresolved — they're strings. Types are unknown. Analyzed logical plan. The Analyzer runs a batch of rules that resolve attributes against the Spark catalog ( SessionCatalog + ExternalCatalog ), binds types, expands * projections, resolves function calls, and validates the tree. Output: a tree where every AttributeReference has a concrete ExprId and DataType . Optimized logical plan. The Optimizer runs many rule batches: Operator Optimization , Decimal Optimizations , LocalRelation , Replace Operators , etc. Each rule is a Rule[LogicalPlan] that pattern-matches a sub-tree and returns a rewritten one. Rules run in batches until fixed point or a max-iteration cap. Physical plan. The SparkPlanner maps logical operators to SparkPlan (physical) operators via the Strategy API. One logical Join maps to one of: BroadcastHashJoin , ShuffledHashJoin , SortMergeJoin , BroadcastNestedLoopJoin , CartesianProduct . The choice is driven by side sizes, broadcast threshold, and hints. Rule-based optimization — the families to memorise. Column pruning ( ColumnPruning ). Drops projections you don't need. If the final aggregation uses only user_id and amount , every other column is pruned at the scan. Predicate pushdown ( PushDownPredicates ). Moves Filter operators down the tree, past Project and (when correct) past Join . The lowest landing spot is inside the scan — Parquet / Iceberg / Delta readers can skip row groups using min/max stats. Constant folding ( ConstantFolding ). Evaluates expressions on literals at compile time. 1 + 2 becomes 3 ; length("hello") becomes 5 . Boolean expression simplification ( BooleanSimplification ). x AND TRUE becomes x ; x OR FALSE becomes x ; NOT NOT x becomes x . Projection collapse ( CollapseProject ). Adjacent Project operators are merged into one — df.select("a","b").select("a") becomes df.select("a") . Inferring filters from joins ( InferFiltersFromConstraints ). If a = b after a join and you have a > 10 , Catalyst infers b > 10 and pushes it to the other side. Replace operators. EXCEPT becomes LeftAntiJoin ; INTERSECT becomes LeftSemiJoin . Set operators map to join operators for execution. Cost-based optimization — when stats unlock joins. CBO is off by default — spark.sql.cbo.enabled must be true and spark.sql.cbo.joinReorder.enabled to enable join reorder. ANALYZE TABLE t COMPUTE STATISTICS captures row count + size. Add FOR COLUMNS col1, col2, ... to capture per-column histograms. With histograms, Catalyst's JoinReorderDP can pick the best join order using a dynamic-programming algorithm over join graphs (similar to System R). Without CBO, the optimizer uses heuristics: tiny side → broadcast; otherwise sort-merge. The Strategy API — how logical becomes physical. A Strategy is a function LogicalPlan -> Seq[SparkPlan] . The planner tries each strategy in order; the first non-empty result wins. Built-in strategies include: JoinSelection , Aggregation , BasicOperators , FileSourceStrategy , DataSourceV2Strategy . Users can add a custom strategy via spark.experimental.extraStrategies (rare; mostly for connector authors). Spark catalog — where attributes get resolved. The SessionCatalog is the per-session view of available tables, temp views, functions. It delegates to an ExternalCatalog (Hive metastore or Spark's in-memory metastore) for persistent tables. Iceberg and Delta plug in via Catalog extensions — spark_catalog , iceberg , delta . The Analyzer routes name resolution through them transparently. Common interview probes on Catalyst. "Name three rules in Catalyst." — column pruning, predicate pushdown, constant folding (any three). "What is the difference between unresolved and analyzed plans?" — name binding + type resolution against the catalog. "How does Catalyst pick a join strategy?" — JoinSelection strategy looks at size hints, broadcast threshold, side estimates. "What is whole stage codegen and where does it fit?" — physical-plan-time compilation of adjacent operators into a single JVM method; lives below Catalyst, above execution. Senior interview question on Catalyst rule batches A senior interviewer might ask: "Catalyst rules run in batches. Walk me through what a batch is, why batches exist, what RuleExecutor does between iterations, and why some rules need multiple passes to converge." Solution Using RuleExecutor and fixed-point iteration RuleExecutor — the rule-batch engine ==================================== A Batch is: name + List[Rule] + Strategy A Strategy is: - Once → run each rule in the batch exactly once - FixedPoint(N) → run the batch repeatedly until either: (a) the tree no longer changes between iterations (b) we hit max iterations N (default 100) RuleExecutor.execute(plan): for batch in batches: iter = 0 loop: iter += 1 newPlan = plan for rule in batch.rules: newPlan = rule.apply(newPlan) if newPlan equals plan: break // fixed point reached if batch.strategy is Once or iter >= N: break // bailout plan = newPlan return plan Step-by-step trace. Iteration Rule Tree state change 1 PushDownPredicates Filter moves below Project 1 CollapseProject adjacent Projects merge 1 ColumnPruning unused columns dropped from scan 2 PushDownPredicates new opportunity revealed by pruning 2 CollapseProject another Project merges 2 ColumnPruning no change 3 (all rules) no change → fixed point The iteration count for the Operator Optimization batch is typically 2-3 in practice. Each rule enables others; the batch converges when no rule rewrites the tree anymore. Output: Batch (selected) Strategy What it does Substitution FixedPoint(100) CTE substitution, window expression resolution Resolution FixedPoint(100) analyzer batch — name + type binding Operator Optimization FixedPoint(100) pruning, pushdown, folding, collapse Decimal Optimizations FixedPoint(100) decimal precision rewrites Join Reorder Once DP join reorder (CBO only) LocalRelation FixedPoint(100) const-folded tree → LocalRelation Why this works — concept by concept: Batches group related rules — running CollapseProject once and never again would miss the projects that only become adjacent after ColumnPruning . The batch lets the rules interact. FixedPoint until convergence — guarantees the optimizer doesn't leave low-hanging rewrites on the table just because of rule ordering. Idempotence is what makes Catalyst safe. Max iterations bailout — protects against pathological rules that flip-flop the tree. In practice 100 is never reached; 2-5 is typical. Strategy = Once vs FixedPoint — some passes (join reorder) are intrinsically one-shot; running them in a fixed-point loop would re-explore the same DP space. Rules compose via tree-matching — every rule is a partial function over the tree's shape. Composability is what lets new rules ship without redesigning the optimizer. Cost — each iteration is O(tree-size × rule-count). With 2-3 iterations and ~50 rules on a typical query, that's bounded compile-time work — milliseconds for most queries. Adaptive Query Execution — runtime replanning adaptive query execution is Catalyst's runtime feedback loop — three rewrites (coalesce, skew split, broadcast swap) account for 90% of its real-world wins The mental model in one line: AQE wraps the physical plan in an AdaptiveSparkPlan node, executes one shuffle stage at a time, and re-optimizes downstream stages using the actual shuffle map output statistics — the optimizer at compile time could only guess; AQE knows . Once you internalise "AQE runs Catalyst rules between stages with real numbers," every spark aqe interview question becomes a deduction from "what does the stage stats unlock that compile-time stats cannot?" Why AQE exists — what compile-time cannot know. Shuffle output sizes. The optimizer can estimate the input to a shuffle from table stats, but the actual output depends on per-key cardinality and filtering — only the executors know. Skew distribution. Stats give you average row count per key, not the long tail. The one VIP customer with 30% of all orders is invisible to CBO. Late-binding broadcast eligibility. A scan + filter + agg may shrink one side of a join from "20GB at compile time" to "8MB at runtime." Compile-time chose sort-merge; runtime would have chosen broadcast. Dynamic partition counts. The right post-shuffle partition count depends on the actual shuffle size. 200 partitions for a 200MB shuffle is criminal; 200 partitions for a 200GB shuffle is correct. The AQE wrapper — AdaptiveSparkPlan . When AQE is enabled, the physical plan is wrapped in AdaptiveSparkPlan . The wrapper materialises one shuffle boundary at a time: it executes the leaf stages, collects their MapOutputStatistics , then re-runs a subset of optimizer rules on the downstream plan. The downstream rules are a special set: CoalesceShufflePartitions , OptimizeSkewedJoin , DynamicJoinSelection , EliminateLimits , etc. They run only at AQE-rewrite time, not at compile time. The rewritten plan replaces the not-yet-executed portion of AdaptiveSparkPlan . Execution continues. The three big AQE features. Coalesce post-shuffle partitions. After a shuffle, AQE merges adjacent tiny partitions into bigger ones. spark.sql.shuffle.partitions is the upper bound; AQE reduces it dynamically. Target partition size: spark.sql.adaptive.advisoryPartitionSizeInBytes (default 64MB). Skew join split. AQE detects partitions whose size is > skewedPartitionFactor * median AND > skewedPartitionThresholdInBytes . It splits the skewed partition into smaller sub-partitions on the skewed side and replicates the matching rows on the other side. The downstream join still produces correct results; one slow reducer becomes N fast reducers. Dynamic switch to broadcast hash join. If, after a shuffle, AQE sees that one side of a planned SortMergeJoin is below spark.sql.adaptive.autoBroadcastJoinThreshold (default 10MB), it rewrites the join to BroadcastHashJoin . The shuffled small side is broadcast instead. Config surface — what you actually set. spark.sql.adaptive.enabled=true — master switch (on by default since Spark 3.2). spark.sql.adaptive.coalescePartitions.enabled=true — coalesce feature. spark.sql.adaptive.skewJoin.enabled=true — skew split feature. spark.sql.adaptive.advisoryPartitionSizeInBytes=64m — target size for coalesced partitions. spark.sql.adaptive.skewJoin.skewedPartitionFactor=5 — partition is "skewed" if > 5x the median. spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes=256m — absolute floor for skew detection. spark.sql.adaptive.autoBroadcastJoinThreshold=10m — broadcast threshold for AQE-time decisions only ; separate from the compile-time spark.sql.autoBroadcastJoinThreshold . When AQE actually wins. Big shuffle, no stats. The compile-time plan defaults to sort-merge join with 200 shuffle partitions. AQE coalesces to ~10-20 and may swap to broadcast. Skewed keys you cannot detect at compile time. Hot keys (VIP customers, popular SKUs, country=US) trigger skew split — the long-tail task drops from 20 minutes to 90 seconds. Joins where one side is unpredictable. Filtered or computed inputs whose post-filter size can't be estimated. First-pass jobs on new data. Statistics don't exist yet; AQE substitutes for ANALYZE TABLE . When AQE adds overhead without winning. Already-correct plans. If you've manually set partition counts and broadcast hints to the right values, AQE rewrite is no-op overhead. Tiny queries. Sub-second queries where the planning-rewrite-execute cycle adds proportionally more overhead than the saved work. High-throughput streaming. AQE applies to batch and to Structured Streaming micro-batches; for sub-100ms micro-batches the rewrite cost can dominate. Common interview probes on AQE. "What are the three AQE features?" — coalesce, skew split, broadcast swap. "When is AQE coalesce a no-op?" — when the compile-time partition count is already correct (rare in practice; AQE almost always finds room to coalesce). "How does skew split keep results correct?" — it splits the skewed side into sub-partitions and replicates the matching rows on the non-skewed side; the join result is the union of sub-results. Reading a Spark plan — EXPLAIN, EXPLAIN FORMATTED, DAG UI df.explain("formatted") is the single most useful five-second action — it tells you what Catalyst chose, what AQE will rewrite, and where to look in the DAG UI The mental model in one line: EXPLAIN is the canonical interface for "what is Spark actually about to do?" — read the tree top-down, follow the join strategies, check PushedFilters and PartitionFilters , then jump to the SQL tab in the UI to see the per-stage task distribution . Once you internalise the read order, every "why is my query slow?" question becomes a structured diagnostic instead of a guess. The three EXPLAIN modes — when to use which. df.explain() — the physical plan only , single page. Use this for quick scanning during development. Shows operator names ( SortMergeJoin , BroadcastHashJoin , HashAggregate , Exchange ). df.explain("extended") — prints all four plans in order: Parsed → Analyzed → Optimized → Physical. The best default for production debugging. Reading the plan top-down — the canonical order. Root. What's the final operator? Usually HashAggregate , Sort , Window , or a writer node ( WriteParquet , AppendData ). Joins. Find every Join node. Note the strategy: BroadcastHashJoin (small side broadcast), SortMergeJoin (both sides sorted + merged), ShuffledHashJoin (rare; AQE may produce these), BroadcastNestedLoopJoin (small × small), CartesianProduct (red flag — usually a missing join key). Recipe 1 — Force broadcast: broadcast() hint vs threshold bump. broadcast() hint wraps the small side: large.join(F.broadcast(small), "key") . The hint always wins over the threshold; use it when you know the small side fits and don't want to depend on stats. spark.sql.autoBroadcastJoinThreshold is the compile-time threshold (default 10MB). Bump to 200MB if you have a small dim table that should always broadcast. Higher than 200-500MB risks driver OOM on the broadcast collect. AQE's autoBroadcastJoinThreshold is a separate runtime knob. Set both for consistent behaviour. When the broadcast still doesn't fire : the small side either (a) has no stats (run ANALYZE), (b) is wrapped in a filter the optimizer can't size, or (c) is in an outer-join position the optimizer refuses to broadcast. Recipe 2 — Shuffle partition count: spark.sql.shuffle.partitions vs AQE coalesce. Pre-AQE rule of thumb : partitions ≈ (total shuffle bytes) / 128MB. For a 200GB shuffle, ~1600 partitions. AQE-era rule of thumb : set spark.sql.shuffle.partitions to a high upper bound (1000–2000), let coalescePartitions reduce it. AQE corrects up only if you've coalesced too aggressively; it cannot exceed the configured maximum. Don't set per-query : prefer the cluster default + AQE. Per-query overrides become noise after 50 queries. Recipe 3 — Dynamic partition pruning (DPP) requirements checklist. ✅ Fact table is partitioned by the join column. ✅ Dim side is broadcast-eligible (under threshold or hinted). ✅ Both sides participate in a regular equi-join (not full outer). ✅ spark.sql.optimizer.dynamicPartitionPruning.enabled=true (default since Spark 3.0). ❌ Doesn't fire on partitioned-only sort-merge joins (the broadcast is required to pass values to the partition filter at runtime). Recipe 4 — AQE skew threshold tuning. Default : skewedPartitionFactor=5 , skewedPartitionThresholdInBytes=256MB . Works for most jobs. Heavily skewed : lower the factor to 3 if you have skewed jobs that AQE isn't catching. Long-tail noise : raise the factor to 10 if AQE is splitting too eagerly on jobs with mild skew (rare). The absolute floor matters : a 5×-median partition that's only 50MB is rarely worth splitting. Keep the floor at 256MB unless your jobs are tiny. Recipe 5 — Whole-stage codegen optimisation. WSCG fuses adjacent operators in the same shuffle boundary into a single JVM method. Eliminates per-row interpreter overhead. Python UDFs break WSCG — the row has to leave the JVM, cross the Python interpreter, return. Each UDF call is ~10x slower than a built-in. Pandas UDFs (Arrow) keep WSCG intact at the boundary (batches cross via Arrow) and use vectorised Python. Always prefer Pandas UDFs. Scala / Java UDFs stay in the JVM and fuse cleanly — no overhead. Built-in functions ( F.col , F.when , F.coalesce , etc.) always WSCG. Recipe 6 — Optimizer-aware code patterns. No count() in hot loops — df.count() forces a full action. Use .cache() + .count() once, or .isEmpty() for existence checks. .cache() only when used > 1 . Caching a DataFrame used once is pure overhead. F.expr("...") over Python UDFs when the logic can be expressed in SQL. Filter before join, not after. If you must filter post-join, ensure the predicate is pushdown-friendly (no UDFs, no lit() wrapping). select the columns you need early. Catalyst will prune anyway, but explicit select makes the intent obvious to the next reader. Recipe 7 — Stats hygiene. Run ANALYZE TABLE t COMPUTE STATISTICS FOR ALL COLUMNS after every large write that changes cardinality by > 20%. Cheap for partitioned tables: ANALYZE TABLE t PARTITION (date='2026-06-22') COMPUTE STATISTICS FOR ALL COLUMNS updates just the new partition. Enable CBO in cluster defaults ( spark.sql.cbo.enabled=true , spark.sql.cbo.joinReorder.enabled=true ). For Delta / Iceberg, the table metadata may already include stats — verify with DESCRIBE EXTENDED t . Senior interview talking points (what to volunteer, unprompted). The Catalyst optimizer is Spark's query planner — the compiler that turns your DataFrame or SQL into an executable physical plan. It runs in four stages: (1) parse the user code into an unresolved logical plan; (2) analyze it by resolving columns and types against the Spark catalog; (3) optimize the logical plan by applying rule batches (column pruning, predicate pushdown, constant folding, projection collapse, boolean simplification, and — with CBO — join reorder); (4) plan the physical operators ( SortMergeJoin , BroadcastHashJoin , HashAggregate , etc.) via the Strategy API. Each rule is a Rule[LogicalPlan] that pattern-matches a tree fragment and returns a rewritten one; rules run in batches until the tree stabilises. The spark catalyst optimizer is what makes Spark's DataFrame API competitive with hand-tuned SQL — but it can only optimise with the information it has at compile time, which is why spark aqe exists. When does AQE actually help? adaptive query execution actually helps whenever the compile-time planner couldn't predict a real-world property of the data: shuffle output sizes, post-filter cardinalities, key distributions. The three big wins: (1) coalesce post-shuffle partitions — after a filtering aggregation, AQE merges tiny partitions so the downstream stage runs N tasks instead of 200; (2) skew join split — splits a partition where one key has 30% of the data into multiple sub-partitions and replicates the matching small side, turning a 22-minute long-tail task into a 90-second one; (3) dynamic switch to broadcast hash join — if a filtered side comes in below the broadcast threshold at runtime, AQE rewrites a planned SortMergeJoin to BroadcastHashJoin, skipping a full shuffle. AQE adds zero overhead on tiny queries and meaningful overhead on sub-100ms streaming micro-batches; for batch jobs it's almost always a win. It's on by default in Spark 3.2+. How do I force a broadcast join in Spark? Two ways. (1) Add a broadcast() hint to the small side: large.join(F.broadcast(small), "key") — this always picks BroadcastHashJoin regardless of size or statistics. Use this when you want per-query determinism and you know the small side fits in driver + executor memory (rule of thumb: ≤ ~500MB total). (2) Bump spark.sql.autoBroadcastJoinThreshold (default 10MB) for the session: spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "200m") . This affects every join in the session; risky if any unexpectedly-large table sneaks under it. (3) Let AQE do it at runtime — with spark.sql.adaptive.autoBroadcastJoinThreshold set, AQE swaps a planned SortMergeJoin to BroadcastHashJoin when post-shuffle the small side is below the threshold. Production-grade pipelines often combine all three: AQE as the safety net, hint for known-small dims, threshold bump only when 50%+ of queries benefit. If the broadcast still doesn't fire, check EXPLAIN FORMATTED for BroadcastHashJoin vs SortMergeJoin and look at the Build side field. What is dynamic partition pruning? spark dynamic partition pruning (DPP) is a Catalyst optimisation that propagates a filter on one side of a star-join to the partition filter on the other side, even when the original predicate wasn't on the partition column. It fires when: (a) the fact table is partitioned by the join column; (b) the dim side is broadcast-eligible (small enough to broadcast or hinted); (c) it's an equi-join. At runtime, the broadcast values become a dynamicpruningexpression in the fact's PartitionFilters , so the fact scan only reads the matching partitions. Example: fact_sales partitioned by sale_date , joined to a dim_dates filtered to "last 7 days" — without DPP, scan all 3650 partitions; with DPP, scan only 7. Verify in EXPLAIN FORMATTED by looking for PartitionFilters: [dynamicpruningexpression(...)] on the partitioned scan. Enabled by default since Spark 3.0 via spark.sql.optimizer.dynamicPartitionPruning.enabled . How do I read a Spark EXPLAIN plan? Use df.explain("formatted") as your default — it prints a compact tree plus a side table of operator details.
