Why vector DBs landed in the DE stack in 2026 A vector database is an ANN index plus a filter engine plus metadata — not "a database that stores arrays" The one-sentence invariant: a vector database is the marriage of an approximate-nearest-neighbour (ANN) index, a typed metadata filter engine, and durable storage — so that "find the 10 most similar embeddings where tenant_id = 42 and status = 'published' " answers in single-digit milliseconds at billions of rows . Everything else — the API shape, the index parameter names, the pricing model — follows from that one definition. Once you internalise "ANN + filter + metadata," the entire vector db comparison interview surface collapses to a series of consequences from how each engine implements those three primitives. Three primitives interviewers actually probe. ANN index. The index is the heart of the system; it answers "given a query vector, return the K closest stored vectors" in sub-linear time. The dominant choice in 2026 is HNSW (Hierarchical Navigable Small World graphs), with IVF (inverted-file) and DiskANN as the alternatives for memory-constrained deployments. Exact O(N) scan is only viable below ~100K vectors. Filter expressiveness. Real workloads never query the whole corpus; you want "the closest 10 documents for this tenant , in this language , published after 2025-01-01 ." How the engine pushes filters into the ANN traversal (pre-filter vs post-filter vs interleaved) drives recall, latency, and correctness. Freshness. Embeddings are not static. A new document arrives; you need it queryable within seconds. The ingest path — write to storage, write to index, make atomically visible — varies wildly between engines and is the most common production bug in 2026. The four engines you must be ready to compare. Pinecone — fully managed, serverless, opinionated. Storage and compute decoupled in Serverless; you pay per-query and per-stored-vector. Zero operations; tight integration with OpenAI / Anthropic embedding APIs. Weaviate — open-source, schema-rich. Classes with typed properties, built-in vectorisers (text2vec-openai, multi2vec-clip), modules for hybrid search and generative answers. Self-host or Weaviate Cloud. Qdrant — open-source, Rust-native, ANN-first. HNSW + scalar / product quantization, payload filters with a rich condition syntax, sparse-vector + dense-vector hybrid. Designed for high-recall ANN at the lowest memory footprint. pgvector — Postgres extension. Adds a vector(N) column type plus HNSW and IVFFlat indexes. Inherits everything Postgres already gives you: ACID, joins, foreign keys, partitioning, replication. The "your warehouse already has a vector DB" pattern. The 2026 reality — what changed since 2023. Pinecone Serverless decoupled storage from compute, making it economical to keep multi-billion-vector corpora without paying for always-on pods. Weaviate 1.24+ ships native multi-tenant collections, async indexing, and a much better hybrid search reranker. Worked example — exact vs approximate nearest neighbour on a 1M-vector corpus Detailed explanation. Before any index, the question is "do you actually need ANN?" An exact O(N) scan on 1 million 768-dimensional vectors is roughly 1M × 768 × 4 bytes = 3 GB of memory bandwidth per query. At ~30 GB/s memory bandwidth, that is ~100 ms per query — borderline for a single user, catastrophic for a thousand QPS. ANN trades a small recall loss (typically 95–99% recall@10) for a 10–100x speedup. Below ~100K vectors, exact scan is often fine and saves you an index entirely. Question. Given a corpus of 1,000,000 vectors at 768 dimensions, compare exact scan vs an HNSW ANN index for top-10 cosine similarity. Show the latency math and the recall trade-off. Input. Parameter Value Corpus size N 1,000,000 Dimension d 768 Query QPS 500 Latency budget 50 ms p99 Recall target 95% recall@10 Code. import numpy as np from hnswlib import Index # Exact scan baseline def exact_topk ( corpus : np . ndarray , query : np . ndarray , k : int = 10 ): sims = corpus @ query # cosine assumes both normalised return np . argpartition ( - sims , k )[: k ] # HNSW index — build once, query many idx = Index ( space = " cosine " , dim = 768 ) idx . init_index ( max_elements = 1_000_000 , ef_construction = 200 , M = 16 ) idx . add_items ( corpus , ids = np . arange ( 1_000_000 )) idx . set_ef ( 64 ) # query-time recall knob labels , distances = idx . knn_query ( query , k = 10 ) Step-by-step explanation. The exact scan multiplies the query against every stored vector — 1,000,000 × 768 × 4 bytes = 3 GB of FLOPs per query. At ~30 GB/s, that is roughly 100 ms per query on commodity hardware; SIMD intrinsics get you to ~30 ms but you have already blown the 50 ms budget at any concurrency. The HNSW index walks a multi-layer graph: starting from a top-layer entry point, it greedily moves toward the query, descending one layer at a time. Each query touches ~ O(log N) × ef neighbours instead of all N — for N = 1M , that is ~ 20 × 64 = 1280 distance computations per query. ef_construction controls build-time graph quality (higher = better recall, slower build); M is the max edges per node (typical 16); ef (query-time) is the candidate-list size and is the main recall knob in production. At ef = 64 , HNSW returns 95%+ recall@10 on the 1M corpus in ~1–2 ms per query — about 50x faster than the exact scan, with the 5% recall loss usually invisible to end users. The trade-off is build time and memory: HNSW costs ~ (M + 1) × N × 8 bytes for the graph (≈ 150 MB on a 1M corpus) plus the vectors themselves. IVF is cheaper to build but slower to query at the same recall. Output. Reach for IVF only when memory pressure forces it. Worked example — pre-filter vs post-filter on a tenant query Detailed explanation. A multi-tenant RAG app stores documents from 10,000 tenants in one corpus. Every query has the implicit predicate tenant_id = ? . Two strategies: pre-filter (eliminate non-matching IDs before ANN) or post-filter (run ANN over the whole corpus then drop non-matching rows). Both are correct in the limit, but the latency and recall differ by orders of magnitude. Question. Given a 100M vector corpus across 10,000 tenants (average 10,000 vectors per tenant), compare pre-filter and post-filter strategies for the query "top 10 nearest for tenant_id = 42 ." Input. Parameter Value Total vectors 100,000,000 Tenants 10,000 Avg vectors per tenant 10,000 Query k 10 Recall target 95% recall@10 within tenant Code. # Pre-filter — Qdrant / Weaviate / pgvector style results = client . search ( collection = " docs " , query_vector = q , query_filter = { " must " : [{ " key " : " tenant_id " , " match " : { " value " : 42 }}]}, limit = 10 , ) # Engine walks ANN graph but skips edges whose payload fails the filter. # Post-filter — naive approach (DO NOT do this for large corpora) candidates = client . search ( collection = " docs " , query_vector = q , limit = 10_000 ) filtered = [ c for c in candidates if c . payload [ " tenant_id " ] == 42 ][: 10 ] # If tenant_id=42 has 0.01% of corpus, you fetch 10K and keep ~1. Step-by-step explanation. Post-filter is the naive approach: run ANN over the whole 100M-vector graph, then discard candidates whose tenant_id != 42 . If tenant 42 is 0.01% of the corpus, the engine has to return ~10,000 candidates to find 10 matches. Recall collapses if your over-fetch is too small. Pre-filter rewrites the ANN traversal: at every graph step, candidates whose payload fails the filter are skipped. The engine walks fewer edges and returns only matching candidates. Qdrant, Weaviate, and pgvector all support this; Pinecone serverless does it via metadata indexes. For high-cardinality filters (10K tenants), pre-filter is faster and recall-preserving. For low-cardinality filters (e.g. language = 'en' matching 90% of corpus), pre-filter and post-filter converge. The deep failure mode: some engines (older Weaviate, naive HNSW implementations) silently degrade recall under pre-filter because the graph traversal can get "stuck" in a region with no matching neighbours. Modern engines mitigate with filter-aware HNSW or partitioned indexes per filter key . The right defensive answer in interviews: "I'd pre-filter when the cardinality of the filter key is high enough that the filtered subset is < 10% of the corpus; otherwise post-filter is fine." Pinecone — managed, serverless pinecone is the managed-by-default vector DB — Serverless decouples storage from compute so you pay for vectors at rest and queries on demand The mental model in one line: a Pinecone index is a logical container of vectors and metadata, partitioned across many internal shards, where (in Serverless mode) storage lives in object storage and compute spins up on demand per query — you never see a node, a shard, or a rebalance event . Once you say "Pinecone abstracts every cluster primitive away," the entire pinecone interview surface becomes a deduction from "what do you still own as a data engineer (schema, namespaces, filters) vs what does Pinecone own (index, replicas, recovery)?" The Pinecone deployment shapes — pods vs serverless. Pods (legacy). You provision a fixed-size pod ( p1 , s1 , p2 with replicas and shards) and pay 24/7. Throughput is bounded by the pod tier. Sizing is your problem. Serverless (2024+). Storage in S3-class storage, compute spun up per query. You pay per stored vector (per-month) plus per read/write unit. No sizing, no pod tier; ideal for spiky workloads and multi-billion-vector corpora. Migration story. Pods → Serverless is a one-way migration via Pinecone's data-export + re-ingest flow. Most new projects start on Serverless. The core abstractions. Index. The top-level container. Has a fixed dimension and metric (cosine, dotproduct, euclidean). All vectors inside the index share these. Namespace. A logical partition inside an index (think "schema" in Postgres). Common pattern: one namespace per tenant or per environment. Queries are scoped to one namespace at a time; cross-namespace queries are not supported. Vector record. A (id, values[d], metadata) triple. Metadata is a JSON blob with typed fields used for filtering. Metadata filters. Pre-filter expressions like {"language": {"$eq": "en"}, "date": {"$gte": "2025-01-01"}} evaluated during the ANN traversal. The 2025–2026 capability additions. Pinecone Inference. Hosted embedding endpoints (Pinecone's own and partner models) so you can embed-and-upsert in a single call instead of round-tripping to OpenAI. Pinecone Assistants. A managed RAG service on top of an index — you upload PDFs, Pinecone chunks + embeds + indexes them, and exposes a chat endpoint. Useful for prototypes; usually replaced with custom RAG once you need control. Hybrid (sparse + dense) search. Each vector record can carry an optional sparse-vector field. Pinecone's hybrid query combines BM25-style sparse scoring with dense ANN scoring at query time. Backup + replicas. Serverless takes continuous incremental snapshots. You can clone an index into a new region in minutes. Pricing model (Serverless, 2026). Stored vectors. Charged per GB-month based on dimension and metadata size. Read units. Charged per query (varies with topK and metadata-filter complexity). Write units. Charged per upsert (rounded up by record size). Free tier. Generous enough for prototyping; production starts ~$30/mo for a small index. The Serverless query path — what happens under the covers. Your client sends an HTTPS POST to the regional Pinecone endpoint with (index, namespace, vector, topK, filter) . The control plane routes the request to a query coordinator for the namespace's shards. Coordinator fans out to all shards owning data for the namespace; each shard fetches its slice from object storage into a hot in-memory cache (or hits the cache directly on subsequent queries). Each shard runs ANN locally with the filter pushed in; returns its local top-K. Coordinator merges the shard-level top-K results and returns the global top-K to the client. Latency is dominated by cold-cache reads from object storage; warm queries are typically < 30 ms. Common interview probes on Pinecone. "What changes when you move from pods to serverless?" — storage decouples from compute; you stop sizing pods; cost shifts to a usage-based model. "What is a namespace and when do you use it?" — logical partition inside an index, one per tenant or per environment; queries scoped to one namespace. "How does Pinecone filter against metadata?" — pre-filter pushed into the ANN traversal during shard-level execution. "How do you handle freshness with Pinecone?" — CDC from your source-of-truth into a worker that upserts; never embed inline. Weaviate — schema-rich, open-source weaviate is the schema-first vector DB — classes with typed properties, built-in vectorisers, modules for hybrid and generative search The mental model in one line: a Weaviate instance hosts a strongly-typed schema (classes with named, typed properties) where each class can plug in a "vectoriser" module that auto-embeds text, images, or multi-modal payloads on insert — and a separate set of modules layers hybrid search, reranking, and generative answers on top of the same index . Once you say "Weaviate is a vector DB with a schema and pluggable modules," the weaviate interview surface becomes a sequence of consequences from that one design choice. The Weaviate core abstractions. Class. Analogous to a SQL table — a named container with a fixed schema of properties. A class has exactly one vector per object (the auto-computed embedding), plus optional named vectors for multi-modal use. Property. A typed field on a class — text , int , boolean , date , geo , cross-reference . Properties are indexed for filtering and BM25 ranking. Object. A row inside a class — (uuid, properties, vector) . The vector is auto-computed by the class's vectoriser on insert/update. Vectoriser (module). A plugin that computes the embedding from the object's properties — text2vec-openai , text2vec-cohere , text2vec-transformers , multi2vec-clip , img2vec-neural , etc. The vectoriser is configured per class. The module ecosystem — what plugs into the index. Vectorisers. text2vec-openai , text2vec-cohere , text2vec-huggingface , text2vec-transformers (self-host), multi2vec-clip (image + text in one vector). Generative. generative-openai , generative-cohere , generative-anthropic — RAG answer generation directly from a Weaviate query. Reranker. reranker-cohere , reranker-transformers — runs a heavier cross-encoder model on the top N ANN results to re-order them by true semantic relevance. Hybrid search. Built-in BM25 + vector hybrid via the hybrid query operator with a tunable alpha . The HNSW + filter implementation. Weaviate uses HNSW as the default ANN index, with parameters efConstruction , maxConnections , ef exposed in the class config. Filters are pre-filtered with filter-aware HNSW : the graph traversal evaluates the filter at every neighbour expansion, skipping objects that fail the predicate. Qdrant — Rust-native, ANN-focused qdrant is the Rust-native vector DB — HNSW with optional quantization, payload-aware pre-filtering, sparse + dense hybrid, the lowest-cost-per-vector self-host The mental model in one line: a Qdrant collection is a set of points (vector + payload) stored in a Rust process that puts every primitive — HNSW build, payload pre-filter, quantization, sparse-vector hybrid — under tight, knob-by-knob control, so a self-hosted Qdrant cluster delivers the lowest cost per stored vector at high recall . Once you say "Qdrant is ANN-first, Rust-native, knob-rich," the qdrant interview surface becomes a sequence of consequences from "every decision is exposed and tunable, but you also own every decision." The Qdrant core abstractions. Collection. Named container with a fixed vector configuration ( size , distance ). One collection per logical dataset (per content type or per tenant for strict isolation). Point. A (id, vector, payload) triple. Payload is a JSON document used for filters and stored alongside the vector. Segment. Internal storage unit. A collection has many segments; each holds a slice of points plus its own HNSW index. Segments rotate (similar to LSM SSTables). Shard. Horizontal scale unit for distributed mode. A collection has N shards; each shard owns a subset of points. Cluster mode replicates shards across nodes for HA. Why Rust matters in production. No GC pauses. HNSW queries hit predictable sub-millisecond latency at the tail because there is no JVM stop-the-world. Predictable memory. RAM usage maps directly to vectors × dim × byte-size; no surprise heap growth. Single binary deploy. No JVM tuning, no JNI bridge, no GIL — just a Rust binary on Linux. Operations become "deploy a service" rather than "tune a runtime." Index + quantization options. HNSW (default). Same algorithm as Weaviate / Pinecone. Enabled per-collection. Product quantization. Each vector chunked and quantized — up to 16x memory reduction with 3–5% recall loss. Used when memory is the binding constraint. On-disk HNSW. The HNSW graph itself lives on disk (mmap), only the working set in RAM. Multi-billion-vector deployments at modest hardware cost. Payload + filter expressiveness. Payload is a JSON document — arbitrary nested structure. Frequently asked questions What is a vector database? A vector database is the marriage of three primitives: an approximate-nearest-neighbour (ANN) index over high-dimensional embeddings, a typed metadata filter engine, and durable storage with a freshness pipeline. You store millions or billions of vectors (typically embeddings from a text or image model) alongside structured metadata (tenant_id, language, date), then run queries like "the 10 most similar embeddings where tenant_id = 42 and lang = 'en' " in single-digit milliseconds. The vector database market in 2026 is dominated by four engines — Pinecone (managed, serverless), Weaviate (schema-rich, open-source), Qdrant (Rust-native, ANN-focused), and pgvector (Postgres-native) — each making different trade-offs across managed-vs-self-host, index type, filter expressiveness, and scale envelope. The most common interview mistake is calling it "a database that stores arrays" — the index + filter combination is the whole product. Pinecone vs Weaviate vs Qdrant — which should I pick? It depends on three answers. Pinecone wins when you want zero operations, predictable usage-based pricing, and a multi-billion-vector ceiling — its Serverless tier decouples storage and compute, so you pay for stored vectors at rest and reads on demand. Weaviate wins when you want a schema-rich, batteries-included engine — typed properties per class, built-in vectorisers (text2vec-openai, multi2vec-clip), hybrid search modules, reranker modules, and generative modules that turn it into a one-call RAG endpoint. Qdrant wins when you self-host and care about lowest cost per stored vector — it is Rust-native (no GC), supports scalar / product quantization for 4–16x memory savings, and ships native sparse + dense hybrid via Reciprocal Rank Fusion. The vector db comparison framework: pick Pinecone for managed simplicity, Weaviate for schema and generative depth, Qdrant for self-host efficiency. Can pgvector replace a dedicated vector database? For corpora up to ~100 million vectors and any team where Postgres is already the source of truth, yes — pgvector postgres is the right answer. You inherit ACID transactions (no separate embedding-pipeline transactional fence), standard SQL joins (no external lookup pipeline), foreign keys, declarative partitioning, streaming replication, and point-in-time recovery — everything Postgres already gives you. Pgvector 0.7+ adds HNSW (alongside the older IVFFlat) and parallel index build, so it competes with dedicated engines on recall and query latency at moderate scale. The scale walls are real: single-node write master limits you to ~10K upserts/sec on tuned hardware, and storage cost grows linearly without quantization. Plan to switch to a dedicated engine when you cross ~200M vectors with sustained high write throughput, or when sub-millisecond cold-cache p99 becomes mandatory. Until then, pgvector deletes a whole tier of infrastructure. What is HNSW and why does it matter for vector databases? HNSW (Hierarchical Navigable Small World) is the dominant ANN index in 2026 — and the vector index hnsw line in every senior interview. It builds a multi-layer proximity graph where the top layer has a few long-range edges and the bottom layer has dense local edges. Queries start at the top layer's entry point, greedily walk toward the query vector, then descend one layer at a time, refining the candidate list at each level. The result: O(log N × ef) distance computations per query instead of O(N) , with 95–99% recall@10 on real workloads. The tunable knobs are m (max edges per node, controls graph density), ef_construction (build-time candidate list, higher = better recall, slower build), and ef or ef_search (query-time candidate list, the main runtime recall knob). HNSW beats IVF on recall at the same query latency, beats brute-force scan by 10–100x in throughput, and is the default index in Pinecone, Weaviate, Qdrant, and pgvector 0.7+. The cost is memory — HNSW adds (m + 1) × N × 8 bytes for the graph on top of the vectors themselves. Hybrid search — vector plus keyword — when should I enable it? Almost always for production search. Pure vector search misses exact-keyword matches (product SKUs, model numbers, proper nouns, code identifiers); pure keyword search misses paraphrase and semantic similarity ("affordable laptop" should match "cheap notebook"). Hybrid search combines a sparse vector (BM25-style or learnt SPLADE weights) with the dense embedding at query time, blending the two scores via a tunable alpha (or via Reciprocal Rank Fusion). Tune alpha = 0.5–0.7 for keyword-heavy domains (catalog, e-commerce, code search, docs); alpha = 0.8–0.95 for paraphrase-heavy domains (FAQ, chat, semantic search). All four engines support hybrid: Pinecone via sparse-vector fields, Weaviate via the hybrid query operator, Qdrant via named sparse + dense vectors with RRF fusion, pgvector via Postgres FTS ( tsvector ) combined with <=> in SQL. The 2026 answer to "should we ship hybrid?" is "yes, by default, then A/B test the alpha." How often should I re-compute embeddings for a vector database? Two rules. Per-document freshness — re-embed any document whose content changes; this is the CDC pipeline (Postgres → Debezium → Kafka → embedding worker → vector DB upsert), end-to-end latency typically < 30 seconds. Never call the embedding API inside the transactional write path — an LLM provider outage would turn into your write outage. Whole-corpus refresh — re-embed every document only when you switch embedding models (e.g., upgrading from text-embedding-3-small to a newer model). Model switches are big events: re-embed in a side index, dual-query during the cutover window, swap traffic to the new index once recall is validated, retire the old index. Day-to-day, embeddings are write-once-per-document; the cost lever is the LLM call, not the vector DB itself.
