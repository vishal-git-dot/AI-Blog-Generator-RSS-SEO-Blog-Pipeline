# The Architecture That Almost Shipped: A Builder Reflection on Deadlines, Cost Anxiety, and What Survives Both

By Ev3lynx · CTO · Builder · AI Systems Architect The competition closed at midnight. My build was green at 11:47. I never hit submit. That terminal window is still in my session history. The browser tabs: YouTube Music on lo-fi, the RSVP page for #JuaraVibeCoding's kick-off — 5,183 people signed up — and a pyproject.toml I'd been polishing for three weeks. I closed the laptop. I'd submit in the morning. By morning, the deadline had passed. This is the story of Ghostclaw — a hybrid Python/TypeScript systems architecture built for Google Indonesia's #JuaraVibeCoding 2026. Designed, built, production-tested. Never submitted. Ships every day. This is not a postmortem. It's a reminder that what survives the deadline is often more valuable than what meets it. The Context In May 2026, Google Indonesia launched #JuaraVibeCoding — a nationwide study jam promoting vibe coding with Google AI Studio, Gemini, and Cloud Run. The engineering track on May 20 drew 768 RSVPs. Campus events at IPB, Airlangga, and Brawijaya followed through end of May. Top entries received swag and e-wallet rewards. We saw it as the perfect proving ground for an architecture pattern we had been refining across multiple projects — the hybrid module : a single package that ships as a CLI tool, an importable library, and an MCP server for AI agents, all sharing one service layer. The project: Ghostclaw — a code analysis engine, a pluggable plugin system, and an MCP server, rolled into one pip install . What Ghostclaw Actually Is ## Four entry points, one codebase ghostclaw # CLI — analyze, review, report ghostclaw - watcher # Continuous file watcher ghostclaw - compare # Diff-based regression analysis ghostclaw - mcp # MCP server — 14 tools for AI agents Four binaries from a single hatchling build. Each entry point is a click.Group subclass registered via project.scripts . The base Command ABC handles argument parsing, error formatting, and output serialization. Adding a subcommand means writing one class and one line in pyproject.toml . The plugin system uses pluggy with entry-point discovery. Third-party developers extend Ghostclaw without forking — install a separate PyPI package that registers a ghostclaw.plugins entry point: [project.entry-points."ghostclaw.plugins"] my_analyzer = "my_package.plugin" The MCP server exposes 14 tools — code analysis, NTP calibration, IMF economic data, DNS dig, currency conversion. Every tool is a pure function decorated with @mcp.tool() . The TypeScript counterpart ships as a dual ESM/CJS package with submodule exports: { "exports" : { "." : { "import" : "./dist/index.mjs" , "require" : "./dist/index.cjs" }, "./mcp" : { "import" : "./dist/mcp/index.mjs" }, "./watcher" : { "import" : "./dist/watcher/index.mjs" } } } Every subpath individually typed, individually importable, individually tree-shakeable. This was the architecture we wanted to submit as our competition entry. Not as a demo, but as a reference implementation of the hybrid module pattern at production scale. The Conversation That Killed the Submission I remember the exact moment. Wednesday night, 1 AM. I was staring at the environment variables I'd wired into the config loader — every one prefixed with GHOSTCLAW_ . The CLI was passing integration tests. The MCP server was responding to tool calls. linter was clean. And I was calculating. "Okay — Cloud Run, one always-on instance, memory overhead for MCP... maybe 20 dollars a month. Plus Rust build in CI, that's GitHub Actions minutes. Dashboard if I add one..." I reached for a calculator. The voice in my head that plays CFO when the real one is sleeping pushed back: "We don't even know if we'll place. Is this worth the time?" The answer should have been obvious. The MCP server doesn't bind a port. It reads stdin and writes stdout. The CLI runs and exits. Zero database, zero always-on services, zero infrastructure. No database? ✅ Zero No server to keep alive? ✅ Zero MCP runs on stdio? ✅ Zero CI on GitHub Actions? ✅ Zero But I didn't check. I estimated . And estimation without data is just anxiety wearing arithmetic as a disguise. I closed the calculator. I told myself I'd figure it out tomorrow. Tomorrow was June 1. The deadline had passed. That's the part I don't talk about in standups. Not that the code was hard — it wasn't. The architecture was sound. The deploy cost was zero. The thing that killed the submission was a conversation I had with myself at a desk, with a calculator, at 1 AM. The Pattern That Outlived the Deadline The submission window closed. The workshops ended. The architecture settled into daily production use. Within two weeks, we extracted Ghostclaw's structural patterns into reusable skills — documented blueprints that scaffold new projects with the same architecture: hybrid-bot-server-builder — CLI + Telegram/Discord bot, one service layer hybrid-module-esm-builder — TypeScript dual ESM/CJS + submodule exports hybrid-module-py-builder — Python src layout, pluggy plugins, maturin addons Each is a complete project template. Run it, get a working hybrid module with CI, tests, docs, and publish config. No manual wiring. Ghostclaw continues as the reference implementation — the project where all three patterns converge: Python hybrid package, TypeScript dual-ship module, and MCP server, sharing one service layer through protocol-agnostic interfaces. Two downstream projects now use this architecture: snapx-app — hybrid CLI + Telegram bot with provider fallback chains wristworks — TypeScript dual-ship with native Rust via napi-rs The submission never happened. The pattern is in production.
